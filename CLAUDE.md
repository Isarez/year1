# นกฮูกสนุกคิด — เตรียมสอบ ป.1

เว็บแอปสำหรับเตรียมสอบเข้า ป.1 ไม่มี build step (ไฟล์ static ล้วนๆ) deploy ผ่าน GitHub Pages จาก branch `main` ที่ https://isarez.github.io/year1/

ชื่อระบบ/แบรนด์: **Owlkids** — มีโดเมนชื่อ **https://owlkids.net**

## หลักการออกแบบหลัก (ยึดทุก feature/ทุกการตัดสินใจ)

- **Static single page template เสมอ** — ไม่มี build step, ไม่มี framework/bundler (React/Vue/webpack/vite ฯลฯ), ไม่มี backend/server-side code ทุก feature ใหม่ต้องทำงานได้ด้วยไฟล์ static ล้วนๆ (HTML/CSS/JS + asset) เปิดผ่าน GitHub Pages ได้ตรงๆ ถ้าจะใช้ library ภายนอก (เช่น Three.js) ต้อง self-host ไฟล์ไว้ใน repo เอง ห้ามพึ่ง CDN หรือ build tool ใดๆ
- **ธีม/ดีไซน์ทุกส่วนต้องออกแบบให้เหมาะกับเด็กอายุ 5 ขวบ** — สีสดใส ทรงมนน่ารัก ตัวอักษรใหญ่อ่านง่าย ไอคอน/ภาพต้องเข้าใจง่ายด้วยสายตาไม่ต้องอ่านออกเขียนได้คล่อง หลีกเลี่ยงองค์ประกอบที่ดูน่ากลัว/สมจริงเกินไป/ซับซ้อนเกินวัย ก่อนออกแบบ element ใหม่ให้อ้างอิงสไตล์ภาพ/ธีมที่มีอยู่แล้วในแอป (การ์ตูนแบน, สีพาสเทล/สดใส, ไอคอน SVG มน ๆ) ให้ไปในทิศทางเดียวกันเสมอ
- **localStorage คือฐานข้อมูลหลักของระบบทั้งหมด ไม่มี backend/server เก็บข้อมูลผู้ใช้** ทุก feature ใหม่ (progress, ตัวละคร, ของสะสม ฯลฯ) ต้องออกแบบให้เก็บ/อ่านผ่าน `localStorage` เป็นหลัก และคิดเผื่อ merge เข้ากับระบบ export/import ย้ายข้อมูลข้ามอุปกรณ์ที่มีอยู่แล้วด้วยเสมอ

## โครงสร้างไฟล์

- `index.html` — markup ล้วนๆ, อ้างอิง CSS/JS ผ่าน `<link>`/`<script src>`
- `css/style.css` — สไตล์ทั้งหมด
- `version` — ไฟล์ text ธรรมดา (ไม่มีนามสกุล) เก็บเลขเวอร์ชัน เช่น `v1.2.0` — **แก้เลขเวอร์ชันที่นี่เมื่อ release ใหม่** หน้าเว็บดึงค่านี้ผ่าน `fetch('version')` แล้วแสดงใน footer
- `js/data.js` — ชุดข้อมูลคำถาม/ข้อความหลัก: `CATS` (คำถามทุกหมวด), `AR_SENTENCES` (ประโยคเกม AR), `CAT_REQUIRES` (กติกาล็อกหมวด)
- `js/owl-messages.js` — `OWL_MSGS` ข้อความให้กำลังใจของนกฮูก
- `js/app.js` — logic ทั้งหมดของแอป (state, render, quiz flow, AR game, ธีม, เสียง ฯลฯ) โหลดหลังสุดเพราะอ้างอิงตัวแปรจาก data.js/owl-messages.js
- ลำดับการโหลด script สำคัญ: `data.js` → `owl-messages.js` → `app.js` (ห้ามสลับ เพราะ app.js ใช้ตัวแปร global จากสองไฟล์แรก) (ห้ามสลับ เพราะ app.js ใช้ตัวแปร global จากสองไฟล์แรก)
- ไม่มี build step ก็จริง แต่การเปิดไฟล์ตรงๆ ผ่าน `file://` ยังใช้ได้ปกติ (ข้อมูลเป็น `.js` ไม่ใช่ `.json` ที่ต้อง `fetch` ผ่าน server เท่านั้น)
- `IDEA.md` — เก็บไอเดียเกม/หมวดใหม่ที่ยังไม่ได้ทำ แต่ละไอเดียเก็บเป็น **หัวข้อเกม** + **หมวดหมู่** + **รายละเอียดเกม** เสมอ (รายละเอียดเติมทีหลังได้ถ้ายังไม่ได้คิด) **อ่านไฟล์นี้เฉพาะตอนจะเริ่มทำเกมใหม่เท่านั้น** (ไม่ต้องอ่านทุกครั้งที่ทำงานอื่น) และเมื่อนำไอเดียไหนไปทำเป็นเกมจริงแล้ว **ต้องลบไอเดียนั้นออกจากไฟล์ทันที**

## กฎการทำงานในโปรเจคนี้

- **ตอบผู้ใช้เป็นภาษาไทยเสมอ** ไม่ว่าจะถามอะไร (โค้ด/ชื่อตัวแปรเป็นภาษาอังกฤษได้ตามปกติ)
- **ห้าม `git push` เอง** ต้องถามผู้ใช้ก่อนทุกครั้ง แม้จะ commit ไปแล้วก็ตาม เมื่อได้รับคำสั่ง "push" หรือ "deploy" ให้ทำตามขั้นตอนนี้เสมอ:
  1. **สรุปสิ่งที่เปลี่ยนแปลงจาก version ก่อนหน้า** — ไฟล์ไหนเปลี่ยน, เพิ่ม/แก้/ลบอะไร
  2. **สรุป task ที่ต้องทำก่อน push** — เช่น อัปเดต `version`, สร้าง GitHub Release, อัปเดต `README.md`, อัปเดต `LOG.md` changelog (ระบุว่าแต่ละอย่างยังต้องทำหรือทำไปแล้ว)
  3. ถามว่า **"confirm push ไหม?"** — รอผู้ใช้ตอบ **"cf"** เท่านั้น จึงจะ push ได้ ห้าม push ก่อนได้รับ "cf" เด็ดขาด
- **ถ้าจะทำ feature ใหม่ ต้องสร้าง branch `feature/<ชื่อ>` แยกจาก `main` ก่อนเริ่มเขียนโค้ดเสมอ** พัฒนา+เทสทั้งหมดบน branch นั้น ห้าม commit ตรงเข้า `main` ระหว่างพัฒนา feature ใหม่ เมื่อทำและเทสเสร็จแล้วให้ **ถามผู้ใช้ก่อนทุกครั้ง** ว่าจะ merge เข้า `main` ไหม (รอคำยืนยันก่อน ห้าม merge เองโดยไม่ถาม) — งานเล็กน้อย (แก้บั๊กเล็ก, ปรับข้อความ/สี, chore ทั่วไป) ไม่ต้องสร้าง branch แยกก็ได้ ใช้ดุลยพินิจว่า "เป็น feature ใหม่" หรือไม่ ถ้าไม่แน่ใจให้ถามผู้ใช้ก่อน
- **ถ้าทำเกมใหม่ (หมวดใหม่ใน `CATS`) หรือนำไอเดียจาก `IDEA.md` มาพัฒนาเป็นเกมจริง ต้องสร้าง branch `feature/<ชื่อ>` แยกจาก `main` ก่อนเริ่มเขียนโค้ดเสมอ ไม่ต้องใช้ดุลยพินิจ ไม่มีข้อยกเว้น** พัฒนา+เทสทั้งหมดบน branch นั้นจนเสร็จ (รวม syntax check, ทดสอบเล่นเกมจริง) แล้ว **ถามผู้ใช้ก่อนทุกครั้ง** ว่าจะ merge เข้า `main` ไหม รอคำยืนยันก่อนเสมอ ห้าม merge เข้า `main` เอง
- **Visitor badge page_id ใน footer (`index.html`, `.visitor-badge` img src)**: ใช้ `isarez.year1.test` เป็นค่า default ระหว่างพัฒนา/ทดสอบ (กันนับยอด visitor จริงเพี้ยนจากการเทส) แต่ก่อน push/deploy จริงทุกครั้งต้อง **แก้ page_id เป็น `isarez.year1`** (ของจริง ไม่มี `.test`) ก่อน commit แล้วพอ push สำเร็จเสร็จสิ้นแล้ว ให้ **แก้กลับเป็น `isarez.year1.test` ทันที** (แก้ไฟล์ตรงๆ เลย **ไม่ต้อง commit หรือ push**) เพื่อกลับสู่โหมดพัฒนาปกติ — ทำทุกครั้งที่ push ไม่ต้องถามก่อน
- **ตรวจสอบก่อนบอกว่าทำเสร็จ**: รัน JS syntax check กับไฟล์ JS ทั้ง 3 ไฟล์ (เรียงตามลำดับโหลดจริง) ก่อนส่งมอบงานทุกครั้ง เช่น
  ```bash
  node -e "
  const fs = require('fs');
  const files = ['js/data.js','js/owl-messages.js','js/app.js'];
  const combined = files.map(f=>fs.readFileSync(f,'utf8')).join('\n');
  try { new Function(combined); console.log('OK JS syntax'); } catch(e){ console.error('ERROR:', e.message); }
  "
  ```
- **หลัง push/deploy สำเร็จทุกครั้ง**:
  1. สรุปสิ่งที่เปลี่ยนแปลงสั้นๆ ให้ผู้ใช้ทราบ
  2. เขียนสรุปนั้นลงไฟล์ `LOG.md` ด้วย โดยเพิ่มเป็นรายการใหม่ **ไว้บนสุด** ของไฟล์ (รูปแบบ: `- YYYY-MM-DD: สรุปสั้นๆ`) แล้วอัปเดตหัวข้อ "ประวัติการเปลี่ยนแปลง (changelog)" ใน `CLAUDE.md` ให้เป็นรายการล่าสุด 5 รายการเสมอ (ตัดรายการเก่าสุดออกถ้าเกิน 5) และถ้าเปลี่ยนแปลงพฤติกรรม/สถานะสำคัญของแอป ให้อัปเดตหัวข้อ "สถานะปัจจุบันของแอป" ใน `CLAUDE.md` ให้ตรงด้วย
  3. ก่อน push/deploy ทุกครั้ง ให้ตรวจและอัปเดต `README.md` ให้ตรงกับสถานะล่าสุดของแอปด้วย (เช่น ฟีเจอร์ใหม่, หมวดหมู่ใหม่) แล้ว commit `README.md` ไปพร้อมกับ `CLAUDE.md`, `LOG.md` และไฟล์โค้ดที่เปลี่ยนใน push เดียวกัน
  4. **ทุกครั้งที่ release version ใหม่ ต้องเขียนทับไฟล์ `changelog` ที่ root ด้วยเสมอ** (ไฟล์ text ธรรมดาไม่มีนามสกุลเหมือน `version`, ใช้แสดงในหน้า "รายละเอียดการอัพเดท" ของแอปให้ผู้ปกครองอ่าน) — **เก็บแค่ version ล่าสุด version เดียว เขียนทับของเดิมทุกครั้ง ไม่สะสมประวัติ** เขียนเป็นภาษาอ่านง่ายสำหรับผู้ปกครอง **ห้ามใส่ศัพท์ technical** (ชื่อไฟล์/ฟังก์ชัน/CSS class ฯลฯ) รูปแบบ: บรรทัดแรกคือเลข version, บรรทัดขึ้นต้น `## ` คือหัวข้อหมวด (เช่น เกมใหม่, ปรับปรุงหน้าตา, แก้ไขข้อผิดพลาด), บรรทัดขึ้นต้น `- ` คือรายการย่อยในหมวดนั้น (ดู parser `parseChangelog()` ใน `js/app.js`) แล้ว commit ไปพร้อมกับไฟล์อื่นๆ ใน push เดียวกัน
- **หลัง `git push` ทุกครั้ง ต้อง track สถานะ deploy ของ GitHub Pages แล้วสรุปสั้นๆ ให้ผู้ใช้ทราบ**:
  1. รอสักครู่แล้วเช็คด้วย `gh run list --limit 3` (workflow "pages build and deployment") และ/หรือ `gh api repos/Isarez/year1/pages` เพื่อดู `status`/`https_certificate`
  2. ถ้ายังไม่เสร็จ ให้เช็คซ้ำเป็นระยะ (เช่นทุก 60-180 วินาทีด้วย ScheduleWakeup) จนกว่าจะจบ — งานอื่นในเทิร์นนั้นทำต่อได้ระหว่างรอ
  3. เมื่อ deploy จบ ให้สรุปสั้นๆ 1-2 บรรทัดว่า **สำเร็จหรือไม่**:
     - สำเร็จ: ยืนยันด้วยการเช็ค `curl -I` ที่ URL จริงว่าได้ 200 และ `last-modified`/`etag` อัปเดตแล้ว (workflow run บางครั้งค้างสถานะ "queued" ทั้งที่ deploy จริงเสร็จแล้ว ให้ยึด `curl` เป็นหลักถ้าขัดแย้งกัน)
     - ไม่สำเร็จ: ต้องบอก **error message** ที่เจอจาก `gh run view <id> --log-failed` ให้ผู้ใช้ทราบด้วยเสมอ ไม่ใช่แค่บอกว่า "fail" เฉยๆ
  4. **แจ้งสรุปผล deploy (ข้อ 3) ใน Discord channel `1524252244965589082` เสมอ** (ใช้ chat_id นี้กับ Discord reply tool) ไม่ว่าจะสั่ง push มาจากช่องทางไหนก็ตาม (Discord ช่องอื่น หรือ session ปกติ)
- **ถ้าเพิ่มเกม/หมวดใหม่ หรือแก้ไขอะไรที่มีนัยสำคัญ (breaking change, ปรับพฤติกรรมหลักของแอป, โครงสร้างไฟล์ใหม่ ฯลฯ) ต้อง `gh release create` ก่อน push/deploy เสมอ**:
  1. ตั้ง tag แบบ semver (เช่น `v1.1.0` ปรับ minor เมื่อเพิ่มฟีเจอร์/เกมใหม่, `v1.0.1` ปรับ patch เมื่อแก้บั๊ก/ปรับเล็กน้อย) — เช็ค tag ล่าสุดด้วย `git tag --list` หรือ `gh release list` ก่อนตั้งเวอร์ชันถัดไป
  2. เขียน release notes สรุปการเปลี่ยนแปลงเป็นภาษาไทย (ใช้เนื้อหาเดียวกับที่จะเขียนลง changelog ใน `LOG.md`)
  3. รัน `gh release create <tag> --title "<tag>: <สรุปสั้นๆ>" --notes "<รายละเอียด>"` แล้วค่อย push commit ตามปกติ (release อ้างอิง commit ปัจจุบันบน `main` ณ ตอนสร้าง)
  4. **แก้ไฟล์ `version` ที่ root ให้ตรงกับ tag ใหม่เสมอ** (เช่น `v1.3.0`) — ทำพร้อมกับ commit อื่นใน push เดียวกัน ห้ามลืม
  5. งานเล็กน้อย (ปรับ CSS ระยะห่าง, แก้ข้อความ, แก้สีเฉพาะจุด) ไม่ต้องสร้าง release ทุกครั้ง — ใช้ดุลยพินิจว่า "มีนัยสำคัญ" หรือไม่ ถ้าไม่แน่ใจให้ถามผู้ใช้ก่อน
- **icon และ emoji ของเกม/หมวดใหม่ต้องไม่ซ้ำกับที่มีอยู่แล้ว**: ก่อนเพิ่มเกม/หมวดใหม่ ให้ grep หรืออ่าน `CATS` ใน `js/data.js` เพื่อดู emoji ที่ใช้อยู่ทั้งหมด แล้วเลือก emoji ที่ยังไม่มีใครใช้ — ทั้ง `cat.emoji` (icon บนการ์ดและสติกเกอร์ในสมุด) และ icon ที่แสดงใน UI อื่นๆ ห้ามซ้ำกันเด็ดขาด เพราะ emoji เดียวกันในสมุดสติกเกอร์จะทำให้แยกหมวดไม่ออก
- **ป้าย "NEW ✨" บนหน้าเลือกหมวด (`cat.isNew:true` ใน `js/data.js`)**:
  1. **ถ้าเพิ่มเกม/หมวดใหม่ ต้องติด `isNew:true` ให้เสมอ** ไม่ต้องถามก่อน (ทำอัตโนมัติทุกครั้งที่เพิ่มหมวดใหม่)
  2. **ห้ามลบ `isNew:true` ของหมวดเดิม/เวอร์ชันก่อนหน้าโดยไม่ถามก่อนเด็ดขาด** — ก่อนลบต้อง **list รายชื่อหมวดที่จะลบ badge ออกมาให้ผู้ใช้ดูก่อนเสมอ** แล้วรอผู้ใช้ยืนยันก่อนแก้ไฟล์ ห้ามลบเองแม้จะดูสมเหตุสมผลว่า "ไม่ใหม่แล้ว" ก็ตาม
- **การ์ดหมวด (`.cat-card`) ในหน้าหลักทุก section (`#cat-grid`, `#cat-grid-interactive`, `#cat-grid-skill`, `#cat-grid-listen` และ section ใหม่ในอนาคต) ต้องมีขนาดเท่ากันเสมอ** — ห้ามปรับ padding/font-size/min-height ของการ์ดเฉพาะหมวดใดหมวดหนึ่งจนขนาดต่างจากการ์ดอื่น หรือปล่อยให้เนื้อหาในการ์ด (ชื่อยาว, meta ยาว) ดันขนาดการ์ดให้เพี้ยนไปจากการ์ดหมวดอื่น ถ้าจะเพิ่มหมวด/เกมใหม่ ให้ใช้ class `.cat-card` เดิมเสมอ ไม่สร้างสไตล์การ์ดแยกเฉพาะหมวด

- **ธีมกลางวัน/กลางคืน**: ปุ่ม `#theme-toggle` ใน header สลับ `body.night-mode`, จำค่าไว้ที่ `localStorage['p1quiz_theme']`. กลางคืน: ท้องฟ้าเปลี่ยนเป็นสีกรมท่า, พระอาทิตย์ (`.bg-sun`) fade ออกและพระจันทร์ (`.bg-moon`) fade เข้า, เมฆเปลี่ยนสี, ลูกโป่งลอยเปลี่ยนเป็นดาว. ตัวอักษรที่ไม่ได้อยู่บนการ์ดขาว/ครีม (header title, hero, footer, section title อย่าง `.cat-section-title`) ต้องมี override สี `body.night-mode` เพราะสีเริ่มต้น (`--ink`/`--ink-soft` น้ำตาลเข้ม) อ่านไม่ออกบนพื้นหลังกลางคืน
  - ข้อควรระวัง CSS: `@keyframes` ที่ animate `opacity` จะ override static opacity rule เสมอ ไม่ว่าจะ specificity อะไร — ถ้าจะซ่อน/โชว์ element ตามธีม ต้อง scope animation ไว้ใน selector ของธีมนั้นๆ ด้วย ไม่ใช่ใส่ไว้ที่ base rule
  - Grid ที่มีเนื้อหาไม่ยอมหด (เช่น emoji ใหญ่) ต้องใช้ `grid-template-columns:repeat(N,minmax(0,1fr))` ไม่ใช่ `repeat(N,1fr)` เฉยๆ ไม่งั้นจะ overflow ออกนอก panel บนจอมือถือแคบ (`#emoji-picker` เจอปัญหานี้มาแล้ว)
- **เพลงพื้นหลัง**: procedural Web Audio (ไม่ใช่ไฟล์เสียง) มี **5 เพลง** (`MUSIC_TRACKS` ใน `js/app.js` แต่ละเพลงมี bpm ของตัวเอง) เล่นจบเพลงแล้วพัก 2 จังหวะต่อเพลงถัดไปวนเป็น playlist, default เป็น **เปิด** สำหรับผู้ใช้ใหม่ (`localStorage['p1quiz_music'] !== 'off'`) เสียงเบา (`0.025`) ห้ามปรับเสียงดังขึ้นโดยไม่ถาม — โน้ตเพลงใช้ oscillator หลักเป็น sine (โทนเดียวกับ `playCorrect`) ห้ามเปลี่ยนกลับเป็น triangle/square โดยไม่ถาม, เสียงคลิกปุ่ม (`playClick`) เป็น sine E5+E6 และมีแฟนแฟร์ `playCongrats()` ดังพร้อมพลุตอนจบเกมทุกโหมด

- **มินิเกม AR (จับคู่นิ้วมือ)**: หมวด `ar-thai`/`ar-eng` (ต่อประโยค ลากคำใส่ช่องตามลำดับ), `ar-math` (คิดเลข บวก/ลบ 1-2 หลัก ลากการ์ดคำตอบที่ถูกใส่ช่องเดียว), `ar-match`/`ar-match-en` (โยงเส้น ลากจากจุดสัญลักษณ์ซ้ายไปยังจุดคำตอบขวาให้ตรงคู่กัน) และ `ar-count` (หยิบให้ครบ — ดูรายละเอียดแยกด้านล่าง) มีแต่หมวด `ar-count` เท่านั้นที่ถือ `isNew:true` ตอนนี้ (5 หมวดก่อนหน้าลบออกแล้วเมื่อ 2026-07-03 เพราะไม่ใหม่แล้ว) ทำให้ขึ้นป้าย "NEW ✨" บนการ์ดหน้าเลือกหมวด (ลบออกเมื่อผู้ใช้เห็นว่าเกมไม่ใหม่แล้วโดยถามก่อน). หน้าเลือกหมวดแบ่งเป็น 2 section: "คำถาม-คำตอบ" (`#cat-grid`) กับ "การโต้ตอบ" (`#cat-grid-interactive`, มีแต่หมวด type `ar`)
  - **มือการ์ตูน (`drawCartoonHand` ใน `js/app.js`)**: วาดสดบน canvas ทุกเฟรมจาก landmark MediaPipe — รับ **พิกัด pixel ที่ผ่าน temporal smoothing แล้ว** จาก `arDrawLoop` (`arHandSmooth`, lerp 50% ต่อเฟรม ใช้ร่วมกันทั้งมือและ cursor ให้ตรงกันเสมอ — ห้ามแยก smooth คนละจุด และต้อง reset `arHandSmooth=null` เมื่อมือหลุดจากกล้อง) องค์ประกอบ: เงาสติกเกอร์ใต้มือ (silhouette วาดลง canvas สำรอง `_handShadowCv` แล้ว `drawImage` ด้วย alpha เดียว — **ห้ามวาดเงาแยกทีละ shape** เพราะรอยซ้อนจะเข้มเป็นหย่อม), ผิว radial gradient, ปลอกแขนเสื้อฟ้าพาสเทลที่ข้อมือ, เล็บนิ้วกลาง/นาง/ก้อย (เว้นโป้ง/ชี้ที่มีจุด indicator), จุด indicator แบบลูกแก้ว highlight+halo เต้นตามเวลา (โป้งแดง=pinch, ชี้ฟ้า=cursor — สีนี้คงที่ห้ามสลับ)
  - `ar-match` (id เดิม, `lang:'th'`) และ `ar-match-en` (id ใหม่, `lang:'en'`) ใช้ข้อมูลคู่จับคู่ร่วมกันจาก `AR_MATCH_ITEMS` ใน `js/data.js` ซึ่งเป็น object แยกภาษา `{th:[...], en:[...]}` (emoji ตรงกันทุกตำแหน่ง คนละแค่คำแปล) — `buildMatchLevel(cat)` เลือก pool ด้วย `AR_MATCH_ITEMS[cat.lang || 'th']` ห้ามลบ id `ar-match` เดิมทิ้งเพราะจะทำให้ progress/สติกเกอร์ผู้ใช้เดิมที่ทำหมวดนี้ไปแล้วหาย ให้เพิ่ม id ใหม่แทนเสมอเวลาจะเพิ่มภาษา
  - `ar-match` ใช้ตัวแปร state แยก (`arDragLineFrom`) และ element ต่างหาก (`#ar-match-wrap`, `#ar-match-left`, `#ar-match-right`, `#ar-match-svg`) จาก 2 โหมดแรกเพราะกลไกคือ "ลากเส้นจากจุด A ไป B" ไม่ใช่ "ลากการ์ดไปวางช่อง" — เส้นที่ลากแล้ว/กำลังลากวาดด้วย SVG `<line>` ผ่าน `getBoundingClientRect()` ของจุด (coordinate ระบบเดียวกับ `.ar-cursor`/`.ar-card` คือ page/client coords ตรงๆ ไม่ผ่าน viewBox scaling)
  - **ท่ามือของ `ar-match` ต่างจาก `ar-thai`/`ar-eng`/`ar-math`**: 2 โหมดแรกใช้ "จีบนิ้วเพื่อหยิบ, คลายจีบเพื่อวาง" (pinch-to-grab) แต่ `ar-match` ใช้ "แตะจุดวงกลมแล้วลากได้ทันที ไม่ต้องจีบ" (เริ่มลากเมื่อปลายนิ้วชี้แตะจุดที่ยังไม่จับคู่ โดยไม่ pinch) และ "จีบนิ้วเพื่อยกเลิกการลาก" (pinch = cancel, ไม่ใช่ drop) — การปล่อย/จับคู่สำเร็จเกิดอัตโนมัติเมื่อปลายนิ้วชี้เข้าใกล้จุดฝั่งตรงข้ามที่ถูกต้อง (ดู `updateArCursor` branch `cat.mode==='match'`) ห้ามเปลี่ยนพฤติกรรมนี้กลับเป็น pinch-to-grab โดยไม่ถาม
  - ข้อควรระวัง CSS: `.ar-slot-row`/`.ar-match-wrap` ต้องมี `[hidden]{display:none;}` กำกับเสมอ เพราะ element ที่มี `display:flex` ประกาศไว้ในคลาสอยู่แล้ว จะไม่ถูกซ่อนด้วย HTML attribute `hidden` เฉยๆ (author CSS ชนะ UA `[hidden]` stylesheet เสมอไม่ว่า specificity จะเท่ากันแค่ไหน)
  - **มือถือ (`window.innerWidth < 768`, ดู `isMobileViewport()` ใน `js/app.js`) ไม่ใช้กล้อง AR เลย**: `initHandTracking()` เช็คก่อนบรรทัดแรกว่าเป็นมือถือหรือไม่ ถ้าใช่จะไม่เรียก `getUserMedia`/MediaPipe เลย (ไม่มี prompt ขอสิทธิ์กล้อง) ข้ามไปโหมดลากด้วยนิ้ว/เมาส์ทันที และแจ้งเตือนด้วย `showToast()` (ครั้งเดียว หายไปเอง) — **ห้ามเปลี่ยนกลับไปใช้ `#ar-camera-msg` แบบการ์ดค้างจอสำหรับกรณีมือถือ** เพราะเคยชนซ้อนทับกับ `.ar-hint-chip` อ่านไม่ออก (การ์ด `#ar-camera-msg` แบบเดิมเก็บไว้เฉพาะกรณีกล้องพังบนเดสก์ท็อป/แท็บเล็ตเท่านั้น ผ่าน `catch(err)`)
  - หน้าเลือกหมวด section "การโต้ตอบ (AR)" มีข้อความแนะนำ (`.cat-section-note`) บอกให้เล่นบนแท็บเล็ต/โน้ตบุ๊ก/คอมพิวเตอร์เพื่อใช้กล้องได้เต็มรูปแบบ และเตือนว่ามือถือจะไม่ใช้กล้อง
  - เมื่อ `body.ar-mobile-nocam` (ตั้งใน `startARGame` ตอนเช็คว่าเป็นมือถือ) พื้นหลัง AR view จะโปร่งใส (`#ar-view`/`.ar-bg-layer{background:transparent}`) ให้เห็นท้องฟ้ากลางวัน/กลางคืนของธีมจริงแทนพื้นหลังกรมท่าคงที่เดิม พร้อมซ่อน `.app-header`/`footer` ไปด้วย (สอง element นี้อยู่นอก view sections เป็น persistent chrome เดิมถูกบัง opacity โดย `#ar-view` ทึบแสง พอพื้นหลังโปร่งใสจะเห็นทะลุเข้ามาถ้าไม่ซ่อน) ส่วนขนาดตัวหนังสือ/การ์ดของเกม AR ทั้งหมดมี media query `@media (max-width:480px)` ลดขนาดลงให้เหมาะกับจอมือถือ
  - **`ar-count` ("หยิบให้ครบ") มีพฤติกรรมมือถือ**ต่างจาก 5 หมวด AR อื่นทั้งหมด**: หมวดอื่นข้ามกล้องแล้วให้เล่นด้วยนิ้วลากต่อได้ปกติบนมือถือ (`isMobileViewport()`) แต่ `ar-count` (`cat.desktopOnly:true`) จะ**ล็อกทั้งหมวดบนมือถือไปเลย** เพราะเกมต้องกระจายของหลายชิ้นบนจอ ต้องการพื้นที่จอกว้างระดับแท็บเล็ต/เดสก์ท็อป — `renderHome()` เช็ค `cat.desktopOnly && isMobileViewport()` แล้วโชว์การ์ดแบบล็อก (เทา, ข้อความ "🖥️ เล่นได้บนแท็บเล็ต/คอมพิวเตอร์เท่านั้นนะ") กดแล้วไม่เปิดเกม แค่ toast เตือน, และ `startARGame()` มี guard ซ้ำอีกชั้นกันเข้าเกมทางอื่น. ถ้าจะเพิ่มเกม AR ที่ต้องการพื้นที่จอกว้างแบบนี้อีกในอนาคต ใช้ flag `desktopOnly:true` ซ้ำได้เลย — **ห้ามใช้ flag นี้กับเกมอื่นที่ไม่ได้ตั้งใจล็อกมือถือโดยถามก่อน**
  - กลไก `ar-count`: โจทย์ (เช่น "หยิบแอปเปิ้ล 5 ลูก") สุ่มจาก `AR_COUNT_QUESTIONS` ใน `js/data.js` (บัคเก็ต `easy`/`medium`/`hard` ตามด่าน เหมือน `AR_SENTENCES[lang][wordCount]`) ของที่กระจายบนจอเป็น `.ar-card.ar-card-plain` (คง class `.ar-card` ไว้ให้ selector JS เดิมใช้ต่อได้ แค่ตัดพื้นหลัง/กรอบออกด้วย `.ar-card-plain` ให้เหลือ emoji เปล่าๆ ขนาดใหญ่) ลากไปวางที่ `#ar-count-zone` (ตะกร้าเดียวรับได้หลายชิ้น ไม่ใช่ slot แยกทีละตำแหน่งแบบโหมดอื่น) เมื่อจำนวนของในตะกร้าครบตามโจทย์ (`arGame.zoneCount === targetCount`) จะล็อกตะกร้าไว้ก่อน (`arGame.zoneLocked`) กันของเกินหลุดเข้ามาระหว่างเช็คผล แล้วตรวจว่าทุกชิ้นเป็นประเภทที่โจทย์ขอหรือไม่ ถ้าไม่ถูกจะรีสโครตของเดิมซ้ำ (ไม่สุ่มโจทย์ใหม่) ให้ลองอีกครั้ง

- **หน้าเลือกหมวดมี "โหมดเกม" 3 แบบ** ตอนนี้ แต่ละแบบมี `<section>`/view-switching logic ของตัวเอง: (1) quiz ปรนัย (`quiz-view`, ไม่มี `type` หรือ `type` อื่น, มี `questions` array) (2) AR กล้อง (`ar-view`, `type:'ar'`, มี `levels`) (3) **เกมฝึกทักษะ** (`memory-view`, `type:'skill'`, มี `levels`) — หน้าเลือกหมวดมี 3 section: "คำถาม-คำตอบ" (`#cat-grid`), "การโต้ตอบ (AR)" (`#cat-grid-interactive`), "เกมฝึกทักษะ" (`#cat-grid-skill`, section ใหม่เพิ่มเมื่อ 2026-07-03) `renderHome()` ใช้ `cat.type` ตัดสินว่า grid ไหน/`startQuiz`/`startARGame`/`startMemoryGame` ตัวไหน (ยกเว้น `type:'skill'` + `mode:'shadow'` แยกไป `startShadowGame` — ดู `skill-shadow` ด้านล่าง) — **ถ้าเพิ่ม view ใหม่ ต้องไล่หา `arView.hidden`/`quizView.hidden`/`memoryView.hidden`/`shadowView.hidden` ทุกจุดที่มีการซ่อน view ทั้งหมด (home-btn, switch-child-btn, `startQuiz`/`startARGame`/`startMemoryGame`/`startListenGame`/`startShadowGame` ที่ต้องซ่อน view อื่นทั้งหมดตอนเข้าเกม) แล้วเพิ่ม view ใหม่เข้าไปด้วยเสมอ ไม่งั้นจะมี 2 view ซ้อนกันได้**
  - **`skill-memory` ("จับคู่โดมิโน")**: เกมความจำกดพลิกการ์ด (ไม่ใช้กล้อง เล่นได้ทั้งมือถือ/แท็บเล็ต/เดสก์ท็อป ไม่มี `desktopOnly`) แบ่งจอ 2 ฝั่ง `#memory-col-dots` (การ์ดจุด, อยู่ซ้าย) / `#memory-col-numbers` (การ์ดตัวเลข, อยู่ขวา) ค่าตัวเลข/จุดสูงสุดคือ **12** มี **3 ด่าน** แต่ละด่านสุ่มค่าไม่ซ้ำจำนวน `MEMORY_LEVEL_PAIRS[level-1]` คู่ (ใน `js/data.js`, ด่าน 1-3 = 4,8,12 คู่ตามลำดับ) shuffle ตำแหน่งฝั่งเลขกับฝั่งจุด**แยกกันเป็นอิสระ**ไม่ให้ตำแหน่งบอกใบ้คู่กัน
  - ดีไซน์การ์ด `.memory-card` ขนาด 92×120px (tablet) / 108×140px (`@media min-width:1025px`) ลายหลังการ์ดใช้ `repeating-linear-gradient` ทแยงทับสีพื้น (การ์ดตัวเลขใช้ `var(--cat-color)` สีของหมวดเอง, การ์ดจุดใช้ `var(--thai)` สีฟ้าประจำแอป) พร้อมไอคอนกลางในวงกลมขาวโปร่ง — ห้ามเปลี่ยนกลับไปใช้ flat gradient เดิมโดยไม่ถาม
  - **การ์ดจุดออกแบบเป็นโดมิโนจริง** (`dominoCardHtml()` ใน `js/app.js`): แบ่งครึ่งบน-ล่าง (`.domino-half` × 2 คั่นด้วย `.domino-divider`) แต่ละครึ่งวางจุดได้สูงสุด 6 จุดตามแพทเทิร์นโดมิโนจริง (`DOMINO_PIPS` lookup 0-6 จุด, ตำแหน่งอิงกริด 3×3 ผ่านคลาส `.pip-tl/.pip-tr/.pip-ml/.pip-c/.pip-mr/.pip-bl/.pip-br`) ค่า N ถูกแบ่งเป็น `top=Math.ceil(N/2)`, `bottom=N-top` เสมอ (การันตีทั้งคู่ไม่เกิน 6 เพราะ N สูงสุด 12) — ถ้าจะขยายค่าสูงสุดเกิน 12 ในอนาคตต้องคิดวิธีแบ่งครึ่งใหม่เพราะเกิน 6/6 จะ overflow ครึ่งการ์ด
  - กลไก flip: คลิกการ์ดคว่ำ 1 ใบต่อฝั่ง (`memoryGame.openNumber`/`openDot`) ถ้าเปิดครบ 2 ฝั่งจะเทียบค่าทันที ตรงกัน→ค้างไว้ถาวร (`matched`), ไม่ตรง→ค้าง `mismatch` ไว้บนจอ**ไม่มี delay/timer คว่ำอัตโนมัติ** (`memoryGame.locked=true`) จนกว่าผู้เล่นจะคลิกการ์ดใบใหม่ใบใดก็ได้ ถึงจะคว่ำคู่เก่ากลับก่อน (`resetMismatch()`) แล้วเปิดใบใหม่ที่เพิ่งคลิกทันที — ออกแบบใหม่แทนของเดิมที่ใช้ `setTimeout` ตายตัว (เคยขยับ 800ms→1500ms→3000ms) เพราะเด็กแต่ละคนใช้เวลานับจุดไม่เท่ากัน ให้เวลาเท่าที่ต้องการจริงๆ แทน — CSS ใช้ 3D flip จริง (`perspective`/`transform-style:preserve-3d`/`backface-visibility:hidden`)
  - stars/progress ใช้เกณฑ์ mistakes เดียวกับเกม AR (`0→3, ≤4→2, else→1`) เพื่อความสม่ำเสมอทั้งแอป ไม่มี per-question review (`review-wrap` ซ่อนเสมอเหมือน AR)
  - **`skill-shadow` ("ทายเงา")**: เกมโจทย์+ตัวเลือกใน section เกมฝึกทักษะ (`type:'skill'`, `mode:'shadow'`, 15 ด่าน, เล่นได้ทุกอุปกรณ์) แต่**ใช้ view แยกของตัวเอง `#shadow-view`** ไม่ใช้ `memory-view` — `renderHome()` dispatch พิเศษด้วย `cat.mode==='shadow'` → `startShadowGame()` ใน `js/app.js`. โจทย์คือเงาสีดำของ emoji (CSS `filter:brightness(0)` บน `#shadow-prompt`) เมื่อตอบแล้วจะเฉลยสีจริงด้วยคลาส `.revealed` — **transition filter ต้องอยู่บน `.revealed` เท่านั้น ห้ามย้ายไป base rule** ไม่งั้นตอนขึ้นด่านใหม่เงาจะแว้บสีจริงให้เห็นก่อนค่อยๆ ดำ (เคยเป็นบั๊กมาแล้ว) ตัวเลือกเป็น emoji สีปกติ+ชื่อไทยกำกับ: ด่าน 1-5 = 3 ตัวเลือก, 6-10 = 4, 11-15 = 5 — คลังโจทย์ `SHADOW_ITEMS` ใน `js/data.js` แบ่ง 3 กลุ่ม (animals 18/fruits 12/objects 15 ชิ้น) แต่ละชิ้นมี **shape tag (`s`)** ตัวหลอกเลือกจาก tag เดียวกับคำตอบก่อน (เงาคล้ายกัน ท้าทายขึ้น) ถ้าไม่พอค่อยเติมสุ่มจากกลุ่มเดียวกัน และคำตอบไม่ซ้ำภายใน 1 รอบเล่นผ่าน `pickNoRepeatIdx` (used Set แยกต่อกลุ่ม) **ตอบผิด = นับ mistake, ไฮไลต์เฉลยตัวเลือกที่ถูก + เปิดสีเงา แล้วข้ามไปด่านถัดไปเลย ไม่ให้เลือกซ้ำ** (ตอบถูกค้าง 1.2s / ตอบผิดค้าง 1.6s ให้ดูเฉลย), ดาวใช้เกณฑ์ mistakes เดิม (`0→3, ≤4→2, else→1`)
- **เกมฟังคำ (`listen-view`, `type:'listen'`)**: `listen1`/`listen2` (คำอังกฤษ, คลัง `LISTEN_WORDS`) และ `listen-th1`/`listen-th2` (คำไทย 3-5 ตัวอักษร, คลัง `LISTEN_WORDS_TH` แยก 3 ระดับความยาว) แยก pool ด้วย `cat.lang` ใน `renderListenLevel()` → `prepareListenLevelEn`/`prepareListenLevelTh` — ตัวอักษรไทยที่ยืนเดี่ยวไม่ได้ (สระบน/ล่าง/วรรณยุกต์) **โชว์เป็นตัวอักษรดิบๆ ตรงๆ ไม่มี prefix ใดๆ** (เคยลองใช้ "o" เป็นฐานแล้วเอาออกหลังทดสอบจริงเพราะดูไม่เหมาะ — **ห้ามใส่ "o" กลับโดยไม่ถาม**) — `speakListenWord()` เช็ค voice ภาษาไทยผ่าน `hasThaiVoiceSupport()`/`pickThaiVoice()` เพื่อ set `u.voice` ตรงๆ และโชว์สัญลักษณ์คำใบ้ (`#listen-symbol`) เสริมถ้าตรวจไม่เจอ voice ไทย แต่**ไม่บล็อกการพูดจริง**อีกต่อไป (เคยบล็อกแล้วเจอบั๊กเสียงไม่ออกเพราะตรวจ false negative ได้ง่าย) พร้อมหน่วง `speak()` 30ms หลัง `cancel()` กันบั๊ก Chrome ที่บางทีเรียกติดกันแล้วเงียบเฉยๆ

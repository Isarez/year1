# นกฮูกสนุกคิด — เตรียมสอบ ป.1

เว็บแอปสำหรับเตรียมสอบเข้า ป.1 ไม่มี build step (ไฟล์ static ล้วนๆ) deploy ผ่าน GitHub Pages จาก branch `main` ที่ https://isarez.github.io/year1/

## โครงสร้างไฟล์

- `index.html` — markup ล้วนๆ, อ้างอิง CSS/JS ผ่าน `<link>`/`<script src>`
- `css/style.css` — สไตล์ทั้งหมด
- `version` — ไฟล์ text ธรรมดา (ไม่มีนามสกุล) เก็บเลขเวอร์ชัน เช่น `v1.2.0` — **แก้เลขเวอร์ชันที่นี่เมื่อ release ใหม่** หน้าเว็บดึงค่านี้ผ่าน `fetch('version')` แล้วแสดงใน footer
- `js/data.js` — ชุดข้อมูลคำถาม/ข้อความหลัก: `CATS` (คำถามทุกหมวด), `AR_SENTENCES` (ประโยคเกม AR), `CAT_REQUIRES` (กติกาล็อกหมวด)
- `js/owl-messages.js` — `OWL_MSGS` ข้อความให้กำลังใจของนกฮูก
- `js/app.js` — logic ทั้งหมดของแอป (state, render, quiz flow, AR game, ธีม, เสียง ฯลฯ) โหลดหลังสุดเพราะอ้างอิงตัวแปรจาก data.js/owl-messages.js
- ลำดับการโหลด script สำคัญ: `data.js` → `owl-messages.js` → `app.js` (ห้ามสลับ เพราะ app.js ใช้ตัวแปร global จากสองไฟล์แรก) (ห้ามสลับ เพราะ app.js ใช้ตัวแปร global จากสองไฟล์แรก)
- ไม่มี build step ก็จริง แต่การเปิดไฟล์ตรงๆ ผ่าน `file://` ยังใช้ได้ปกติ (ข้อมูลเป็น `.js` ไม่ใช่ `.json` ที่ต้อง `fetch` ผ่าน server เท่านั้น)
- `IDEA.md` — เก็บไอเดียเกม/หมวดใหม่ที่ยังไม่ได้ทำ แต่ละไอเดียเก็บเป็น **หัวข้อเกม** + **หมวดหมู่** + **รายละเอียดเกม** เสมอ (รายละเอียดเติมทีหลังได้ถ้ายังไม่ได้คิด) **อ่านไฟล์นี้เฉพาะตอนจะเริ่มทำเกมใหม่เท่านั้น** (ไม่ต้องอ่านทุกครั้งที่ทำงานอื่น) และเมื่อนำไอเดียไหนไปทำเป็นเกมจริงแล้ว **ต้องลบไอเดียนั้นออกจากไฟล์ทันที**

## กฎการทำงานในโปรเจคนี้

- **ตอบผู้ใช้เป็นภาษาไทยเสมอ** ไม่ว่าจะถามอะไร (โค้ด/ชื่อตัวแปรเป็นภาษาอังกฤษได้ตามปกติ)
- **ห้าม `git push` เอง** ต้องถามผู้ใช้ก่อนทุกครั้ง แม้จะ commit ไปแล้วก็ตาม เมื่อได้รับคำสั่ง "push" หรือ "deploy" ให้ทำตามขั้นตอนนี้เสมอ:
  1. **สรุปสิ่งที่เปลี่ยนแปลงจาก version ก่อนหน้า** — ไฟล์ไหนเปลี่ยน, เพิ่ม/แก้/ลบอะไร
  2. **สรุป task ที่ต้องทำก่อน push** — เช่น อัปเดต `version`, สร้าง GitHub Release, อัปเดต `README.md`, อัปเดต `CLAUDE.md` changelog (ระบุว่าแต่ละอย่างยังต้องทำหรือทำไปแล้ว)
  3. ถามว่า **"confirm push ไหม?"** — รอผู้ใช้ตอบ **"cf"** เท่านั้น จึงจะ push ได้ ห้าม push ก่อนได้รับ "cf" เด็ดขาด
- **ตรวจสอบก่อนบอกว่าทำเสร็จ**: รัน JS syntax check กับไฟล์ JS ทั้ง 3 ไฟล์ (เรียงตามลำดับโหลดจริง) ก่อนส่งมอบงานทุกครั้ง เช่น
  ```bash
  node -e "
  const fs = require('fs');
  const files = ['js/data.js','js/owl-messages.js','js/app.js'];
  const combined = files.map(f=>fs.readFileSync(f,'utf8')).join('\n');
  try { new Function(combined); console.log('OK JS syntax'); } catch(e){ console.error('ERROR:', e.message); }
  "
  ```
- **หลัง push/deploy สำเร็จทุกครั้ง**:
  1. สรุปสิ่งที่เปลี่ยนแปลงสั้นๆ ให้ผู้ใช้ทราบ
  2. เขียนสรุปนั้นลงไฟล์นี้ (`CLAUDE.md`) ด้วย โดยเพิ่มเป็นรายการใหม่ในหัวข้อ "ประวัติการเปลี่ยนแปลง (changelog)" ด้านล่าง (รูปแบบ: `- YYYY-MM-DD: สรุปสั้นๆ`) และถ้าเปลี่ยนแปลงพฤติกรรม/สถานะสำคัญของแอป ให้อัปเดตหัวข้อ "สถานะปัจจุบันของแอป" ให้ตรงด้วย
  3. ก่อน push/deploy ทุกครั้ง ให้ตรวจและอัปเดต `README.md` ให้ตรงกับสถานะล่าสุดของแอปด้วย (เช่น ฟีเจอร์ใหม่, หมวดหมู่ใหม่) แล้ว commit `README.md` ไปพร้อมกับ `CLAUDE.md` และไฟล์โค้ดที่เปลี่ยนใน push เดียวกัน
- **หลัง `git push` ทุกครั้ง ต้อง track สถานะ deploy ของ GitHub Pages แล้วสรุปสั้นๆ ให้ผู้ใช้ทราบ**:
  1. รอสักครู่แล้วเช็คด้วย `gh run list --limit 3` (workflow "pages build and deployment") และ/หรือ `gh api repos/Isarez/year1/pages` เพื่อดู `status`/`https_certificate`
  2. ถ้ายังไม่เสร็จ ให้เช็คซ้ำเป็นระยะ (เช่นทุก 60-180 วินาทีด้วย ScheduleWakeup) จนกว่าจะจบ — งานอื่นในเทิร์นนั้นทำต่อได้ระหว่างรอ
  3. เมื่อ deploy จบ ให้สรุปสั้นๆ 1-2 บรรทัดว่า **สำเร็จหรือไม่**:
     - สำเร็จ: ยืนยันด้วยการเช็ค `curl -I` ที่ URL จริงว่าได้ 200 และ `last-modified`/`etag` อัปเดตแล้ว (workflow run บางครั้งค้างสถานะ "queued" ทั้งที่ deploy จริงเสร็จแล้ว ให้ยึด `curl` เป็นหลักถ้าขัดแย้งกัน)
     - ไม่สำเร็จ: ต้องบอก **error message** ที่เจอจาก `gh run view <id> --log-failed` ให้ผู้ใช้ทราบด้วยเสมอ ไม่ใช่แค่บอกว่า "fail" เฉยๆ
- **ถ้าเพิ่มเกม/หมวดใหม่ หรือแก้ไขอะไรที่มีนัยสำคัญ (breaking change, ปรับพฤติกรรมหลักของแอป, โครงสร้างไฟล์ใหม่ ฯลฯ) ต้อง `gh release create` ก่อน push/deploy เสมอ**:
  1. ตั้ง tag แบบ semver (เช่น `v1.1.0` ปรับ minor เมื่อเพิ่มฟีเจอร์/เกมใหม่, `v1.0.1` ปรับ patch เมื่อแก้บั๊ก/ปรับเล็กน้อย) — เช็ค tag ล่าสุดด้วย `git tag --list` หรือ `gh release list` ก่อนตั้งเวอร์ชันถัดไป
  2. เขียน release notes สรุปการเปลี่ยนแปลงเป็นภาษาไทย (ใช้เนื้อหาเดียวกับที่จะเขียนลง changelog ใน `CLAUDE.md`)
  3. รัน `gh release create <tag> --title "<tag>: <สรุปสั้นๆ>" --notes "<รายละเอียด>"` แล้วค่อย push commit ตามปกติ (release อ้างอิง commit ปัจจุบันบน `main` ณ ตอนสร้าง)
  4. **แก้ไฟล์ `version` ที่ root ให้ตรงกับ tag ใหม่เสมอ** (เช่น `v1.3.0`) — ทำพร้อมกับ commit อื่นใน push เดียวกัน ห้ามลืม
  5. งานเล็กน้อย (ปรับ CSS ระยะห่าง, แก้ข้อความ, แก้สีเฉพาะจุด) ไม่ต้องสร้าง release ทุกครั้ง — ใช้ดุลยพินิจว่า "มีนัยสำคัญ" หรือไม่ ถ้าไม่แน่ใจให้ถามผู้ใช้ก่อน
- **icon และ emoji ของเกม/หมวดใหม่ต้องไม่ซ้ำกับที่มีอยู่แล้ว**: ก่อนเพิ่มเกม/หมวดใหม่ ให้ grep หรืออ่าน `CATS` ใน `js/data.js` เพื่อดู emoji ที่ใช้อยู่ทั้งหมด แล้วเลือก emoji ที่ยังไม่มีใครใช้ — ทั้ง `cat.emoji` (icon บนการ์ดและสติกเกอร์ในสมุด) และ icon ที่แสดงใน UI อื่นๆ ห้ามซ้ำกันเด็ดขาด เพราะ emoji เดียวกันในสมุดสติกเกอร์จะทำให้แยกหมวดไม่ออก
- **ป้าย "NEW ✨" บนหน้าเลือกหมวด (`cat.isNew:true` ใน `js/data.js`)**:
  1. **ถ้าเพิ่มเกม/หมวดใหม่ ต้องติด `isNew:true` ให้เสมอ** ไม่ต้องถามก่อน (ทำอัตโนมัติทุกครั้งที่เพิ่มหมวดใหม่)
  2. **ห้ามลบ `isNew:true` ของหมวดเดิม/เวอร์ชันก่อนหน้าโดยไม่ถามก่อนเด็ดขาด** — ก่อนลบต้อง **list รายชื่อหมวดที่จะลบ badge ออกมาให้ผู้ใช้ดูก่อนเสมอ** แล้วรอผู้ใช้ยืนยันก่อนแก้ไฟล์ ห้ามลบเองแม้จะดูสมเหตุสมผลว่า "ไม่ใหม่แล้ว" ก็ตาม

## สถานะปัจจุบันของแอป (สำคัญ อย่าย้อนกลับโดยไม่ถาม)

- **ธีมกลางวัน/กลางคืน**: ปุ่ม `#theme-toggle` ใน header สลับ `body.night-mode`, จำค่าไว้ที่ `localStorage['p1quiz_theme']`. กลางคืน: ท้องฟ้าเปลี่ยนเป็นสีกรมท่า, พระอาทิตย์ (`.bg-sun`) fade ออกและพระจันทร์ (`.bg-moon`) fade เข้า, เมฆเปลี่ยนสี, ลูกโป่งลอยเปลี่ยนเป็นดาว. ตัวอักษรที่ไม่ได้อยู่บนการ์ดขาว/ครีม (header title, hero, footer, section title อย่าง `.cat-section-title`) ต้องมี override สี `body.night-mode` เพราะสีเริ่มต้น (`--ink`/`--ink-soft` น้ำตาลเข้ม) อ่านไม่ออกบนพื้นหลังกลางคืน
  - ข้อควรระวัง CSS: `@keyframes` ที่ animate `opacity` จะ override static opacity rule เสมอ ไม่ว่าจะ specificity อะไร — ถ้าจะซ่อน/โชว์ element ตามธีม ต้อง scope animation ไว้ใน selector ของธีมนั้นๆ ด้วย ไม่ใช่ใส่ไว้ที่ base rule
  - Grid ที่มีเนื้อหาไม่ยอมหด (เช่น emoji ใหญ่) ต้องใช้ `grid-template-columns:repeat(N,minmax(0,1fr))` ไม่ใช่ `repeat(N,1fr)` เฉยๆ ไม่งั้นจะ overflow ออกนอก panel บนจอมือถือแคบ (`#emoji-picker` เจอปัญหานี้มาแล้ว)
- **เพลงพื้นหลัง**: procedural Web Audio (ไม่ใช่ไฟล์เสียง) default เป็น **เปิด** สำหรับผู้ใช้ใหม่ (`localStorage['p1quiz_music'] !== 'off'`) เสียงเบา (`0.025`) ห้ามปรับเสียงดังขึ้นโดยไม่ถาม
- **หมวดหมู่ (`CATS`)**: math, thai (ภาษาไทย 1), thai2 (ล็อกจนกว่าจะทำ thai1 จบ), english, behavior, animals, days, iq1-iq4 (ล็อกตามลำดับผ่าน `CAT_REQUIRES`)
- **"days" category**: ค่าตำแหน่งคำตอบถูก shuffle แบบ one-time manual fix ไปแล้ว (ของเดิมตอบ index 0 หมด) — อย่า shuffle ซ้ำอีก
- **พื้นหลังกลางวัน**: ท้องฟ้าสีฟ้า gradient, มีพระอาทิตย์หมุนช้าๆ, เมฆลอย, ลูกโป่งลอย, ประกายดาวระยิบระยับ — เคยมีสายรุ้งแล้วเอาออกตามคำขอผู้ใช้ **ห้ามใส่กลับโดยไม่ถาม**
- **นกฮูกมาสคอต**: มุมขวาล่าง คลิกแล้วสุ่มข้อความให้กำลังใจจาก `OWL_MSGS.cheer` (คนละปุ่มกับสมุดสติกเกอร์ `#sticker-tally-btn`)
- **Favicon/ไอคอน**: `assets/favicon.svg` เป็น source of truth ถ้าแก้ต้อง regenerate PNG ทุกไซส์ด้วย `rsvg-convert -w <size> -h <size> assets/favicon.svg -o assets/<name>.png`
- **ข้อมูลผู้ใช้ทั้งหมดเก็บใน `localStorage` เท่านั้น ไม่มี backend** — footer มีข้อความยืนยันเรื่องนี้ ต้องคงความจริงนี้ไว้ถ้าจะเพิ่มฟีเจอร์ sync/backend ในอนาคต
- **มินิเกม AR (จับคู่นิ้วมือ)**: หมวด `ar-thai`/`ar-eng` (ต่อประโยค ลากคำใส่ช่องตามลำดับ), `ar-math` (คิดเลข บวก/ลบ 1-2 หลัก ลากการ์ดคำตอบที่ถูกใส่ช่องเดียว), `ar-match`/`ar-match-en` (โยงเส้น ลากจากจุดสัญลักษณ์ซ้ายไปยังจุดคำตอบขวาให้ตรงคู่กัน) และ `ar-count` (หยิบให้ครบ — ดูรายละเอียดแยกด้านล่าง) มีแต่หมวด `ar-count` เท่านั้นที่ถือ `isNew:true` ตอนนี้ (5 หมวดก่อนหน้าลบออกแล้วเมื่อ 2026-07-03 เพราะไม่ใหม่แล้ว) ทำให้ขึ้นป้าย "NEW ✨" บนการ์ดหน้าเลือกหมวด (ลบออกเมื่อผู้ใช้เห็นว่าเกมไม่ใหม่แล้วโดยถามก่อน). หน้าเลือกหมวดแบ่งเป็น 2 section: "คำถาม-คำตอบ" (`#cat-grid`) กับ "การโต้ตอบ" (`#cat-grid-interactive`, มีแต่หมวด type `ar`)
  - `ar-match` (id เดิม, `lang:'th'`) และ `ar-match-en` (id ใหม่, `lang:'en'`) ใช้ข้อมูลคู่จับคู่ร่วมกันจาก `AR_MATCH_ITEMS` ใน `js/data.js` ซึ่งเป็น object แยกภาษา `{th:[...], en:[...]}` (emoji ตรงกันทุกตำแหน่ง คนละแค่คำแปล) — `buildMatchLevel(cat)` เลือก pool ด้วย `AR_MATCH_ITEMS[cat.lang || 'th']` ห้ามลบ id `ar-match` เดิมทิ้งเพราะจะทำให้ progress/สติกเกอร์ผู้ใช้เดิมที่ทำหมวดนี้ไปแล้วหาย ให้เพิ่ม id ใหม่แทนเสมอเวลาจะเพิ่มภาษา
  - `ar-match` ใช้ตัวแปร state แยก (`arDragLineFrom`) และ element ต่างหาก (`#ar-match-wrap`, `#ar-match-left`, `#ar-match-right`, `#ar-match-svg`) จาก 2 โหมดแรกเพราะกลไกคือ "ลากเส้นจากจุด A ไป B" ไม่ใช่ "ลากการ์ดไปวางช่อง" — เส้นที่ลากแล้ว/กำลังลากวาดด้วย SVG `<line>` ผ่าน `getBoundingClientRect()` ของจุด (coordinate ระบบเดียวกับ `.ar-cursor`/`.ar-card` คือ page/client coords ตรงๆ ไม่ผ่าน viewBox scaling)
  - **ท่ามือของ `ar-match` ต่างจาก `ar-thai`/`ar-eng`/`ar-math`**: 2 โหมดแรกใช้ "จีบนิ้วเพื่อหยิบ, คลายจีบเพื่อวาง" (pinch-to-grab) แต่ `ar-match` ใช้ "แตะจุดวงกลมแล้วลากได้ทันที ไม่ต้องจีบ" (เริ่มลากเมื่อปลายนิ้วชี้แตะจุดที่ยังไม่จับคู่ โดยไม่ pinch) และ "จีบนิ้วเพื่อยกเลิกการลาก" (pinch = cancel, ไม่ใช่ drop) — การปล่อย/จับคู่สำเร็จเกิดอัตโนมัติเมื่อปลายนิ้วชี้เข้าใกล้จุดฝั่งตรงข้ามที่ถูกต้อง (ดู `updateArCursor` branch `cat.mode==='match'`) ห้ามเปลี่ยนพฤติกรรมนี้กลับเป็น pinch-to-grab โดยไม่ถาม
  - ข้อควรระวัง CSS: `.ar-slot-row`/`.ar-match-wrap` ต้องมี `[hidden]{display:none;}` กำกับเสมอ เพราะ element ที่มี `display:flex` ประกาศไว้ในคลาสอยู่แล้ว จะไม่ถูกซ่อนด้วย HTML attribute `hidden` เฉยๆ (author CSS ชนะ UA `[hidden]` stylesheet เสมอไม่ว่า specificity จะเท่ากันแค่ไหน)

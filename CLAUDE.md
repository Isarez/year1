# นกฮูกสนุกคิด — เตรียมสอบ ป.1

เว็บแอปสำหรับเตรียมสอบเข้า ป.1 ไม่มี build step (ไฟล์ static ล้วนๆ) deploy ผ่าน GitHub Pages จาก branch `main` ที่ https://isarez.github.io/year1/

ชื่อระบบ/แบรนด์: **Owlkids** — มีโดเมนชื่อ **https://owlkids.net**

## หลักการออกแบบหลัก (ยึดทุก feature/ทุกการตัดสินใจ)

- **Static single page template เสมอ** — ไม่มี build step, ไม่มี framework/bundler (React/Vue/webpack/vite ฯลฯ), ไม่มี backend/server-side code ทุก feature ใหม่ต้องทำงานได้ด้วยไฟล์ static ล้วนๆ (HTML/CSS/JS + asset) เปิดผ่าน GitHub Pages ได้ตรงๆ ถ้าจะใช้ library ภายนอก (เช่น Three.js) ต้อง self-host ไฟล์ไว้ใน repo เอง ห้ามพึ่ง CDN หรือ build tool ใดๆ
- **ธีม/ดีไซน์ทุกส่วนต้องออกแบบให้เหมาะกับเด็กอายุ 5 ขวบ** — สีสดใส ทรงมนน่ารัก ตัวอักษรใหญ่อ่านง่าย ไอคอน/ภาพต้องเข้าใจง่ายด้วยสายตาไม่ต้องอ่านออกเขียนได้คล่อง หลีกเลี่ยงองค์ประกอบที่ดูน่ากลัว/สมจริงเกินไป/ซับซ้อนเกินวัย ก่อนออกแบบ element ใหม่ให้อ้างอิงสไตล์ภาพ/ธีมที่มีอยู่แล้วในแอป (การ์ตูนแบน, สีพาสเทล/สดใส, ไอคอน SVG มน ๆ) ให้ไปในทิศทางเดียวกันเสมอ
- **localStorage คือฐานข้อมูลหลักของระบบทั้งหมด ไม่มี backend/server เก็บข้อมูลผู้ใช้** ทุก feature ใหม่ (progress, ตัวละคร, ของสะสม ฯลฯ) ต้องออกแบบให้เก็บ/อ่านผ่าน `localStorage` เป็นหลัก และคิดเผื่อ merge เข้ากับระบบ export/import ย้ายข้อมูลข้ามอุปกรณ์ที่มีอยู่แล้วด้วยเสมอ

## โครงสร้างไฟล์

- `index.html` — markup ล้วนๆ, อ้างอิง CSS/JS ผ่าน `<link>`/`<script src>`
- `css/style.css` — สไตล์ทั้งหมด
- `version` — ไฟล์ text ธรรมดา (ไม่มีนามสกุล) เก็บเลขเวอร์ชัน เช่น `v1.2.0` — **แก้เลขเวอร์ชันที่นี่เมื่อ release ใหม่** หน้าเว็บดึงค่านี้ผ่าน `fetch('version')` แล้วแสดงใน footer
- `js/data.js` — ชุดข้อมูลคำถาม/ข้อความหลัก: `CATS` (คำถามทุกหมวด), `AR_SENTENCES` (ประโยคเกม AR), `CAT_REQUIRES` (กติกาล็อกหมวด)
- `js/owl-messages.js` — `OWL_MSGS` ข้อความให้กำลังใจของนกฮูก
- `js/app.js` — logic ทั้งหมดของแอป (state, render, quiz flow, AR game, ธีม, เสียง ฯลฯ) โหลดหลังสุดเพราะอ้างอิงตัวแปรจาก data.js/owl-messages.js
- ลำดับการโหลด script สำคัญ: `data.js` → `owl-messages.js` → `app.js` (ห้ามสลับ เพราะ app.js ใช้ตัวแปร global จากสองไฟล์แรก) (ห้ามสลับ เพราะ app.js ใช้ตัวแปร global จากสองไฟล์แรก)
- ไม่มี build step ก็จริง แต่การเปิดไฟล์ตรงๆ ผ่าน `file://` ยังใช้ได้ปกติ (ข้อมูลเป็น `.js` ไม่ใช่ `.json` ที่ต้อง `fetch` ผ่าน server เท่านั้น)
- `IDEA.md` — เก็บไอเดียเกม/หมวดใหม่ที่ยังไม่ได้ทำ แต่ละไอเดียเก็บเป็น **หัวข้อเกม** + **หมวดหมู่** + **รายละเอียดเกม** เสมอ (รายละเอียดเติมทีหลังได้ถ้ายังไม่ได้คิด) **อ่านไฟล์นี้เฉพาะตอนจะเริ่มทำเกมใหม่เท่านั้น** (ไม่ต้องอ่านทุกครั้งที่ทำงานอื่น) และเมื่อนำไอเดียไหนไปทำเป็นเกมจริงแล้ว **ต้องลบไอเดียนั้นออกจากไฟล์ทันที**
- `teacher/` — **โหมดคุณครู** (เข้าผ่าน URL `owlkids.net/teacher` เท่านั้น ไม่มีปุ่มจากหน้าหลัก): `teacher/index.html` + `teacher/teacher.js` + `teacher/teacher.css` — หน้าแยกที่ reuse `css/style.css` และธีม/พื้นหลังเดียวกับหน้าหลัก แต่ **JS แยกของตัวเองทั้งหมด ห้ามโหลด `js/app.js` ในหน้านี้** (app.js ผูกกับ DOM ของ index.html แน่นมาก) logo เกมสำหรับครูเลือกอยู่ที่ `assets/icons/teacher/` (12 แบบ) ข้อมูลครูเก็บ localStorage key `owlkids_teacher_profile` / `owlkids_teacher_games` (แยกจากข้อมูลเด็กหน้าหลัก) — ธีมกลางวัน/กลางคืนใช้ key `p1quiz_theme` ร่วมกับหน้าหลักโดยตั้งใจ ให้สลับธีมแล้วตรงกันทั้งสองหน้า

## กฎการทำงานในโปรเจคนี้

- **ตอบผู้ใช้เป็นภาษาไทยเสมอ** ไม่ว่าจะถามอะไร (โค้ด/ชื่อตัวแปรเป็นภาษาอังกฤษได้ตามปกติ)
- **ห้าม `git push` เอง** ต้องถามผู้ใช้ก่อนทุกครั้ง แม้จะ commit ไปแล้วก็ตาม เมื่อได้รับคำสั่ง "push" หรือ "deploy" ให้ทำตามขั้นตอนนี้เสมอ:
  1. **สรุปสิ่งที่เปลี่ยนแปลงจาก version ก่อนหน้า** — ไฟล์ไหนเปลี่ยน, เพิ่ม/แก้/ลบอะไร
  2. **สรุป task ที่ต้องทำก่อน push** — เช่น อัปเดต `version`, สร้าง GitHub Release, อัปเดต `README.md`, อัปเดต `LOG.md` changelog (ระบุว่าแต่ละอย่างยังต้องทำหรือทำไปแล้ว)
  3. ถามว่า **"confirm push ไหม?"** — รอผู้ใช้ตอบ **"cf"** เท่านั้น จึงจะ push ได้ ห้าม push ก่อนได้รับ "cf" เด็ดขาด
- **ถ้าจะทำ feature ใหม่ ต้องสร้าง branch `feature/<ชื่อ>` แยกจาก `main` ก่อนเริ่มเขียนโค้ดเสมอ** พัฒนา+เทสทั้งหมดบน branch นั้น ห้าม commit ตรงเข้า `main` ระหว่างพัฒนา feature ใหม่ เมื่อทำและเทสเสร็จแล้วให้ **ถามผู้ใช้ก่อนทุกครั้ง** ว่าจะ merge เข้า `main` ไหม (รอคำยืนยันก่อน ห้าม merge เองโดยไม่ถาม) — งานเล็กน้อย (แก้บั๊กเล็ก, ปรับข้อความ/สี, chore ทั่วไป) ไม่ต้องสร้าง branch แยกก็ได้ ใช้ดุลยพินิจว่า "เป็น feature ใหม่" หรือไม่ ถ้าไม่แน่ใจให้ถามผู้ใช้ก่อน
- **ถ้าทำเกมใหม่ (หมวดใหม่ใน `CATS`) หรือนำไอเดียจาก `IDEA.md` มาพัฒนาเป็นเกมจริง ต้องสร้าง branch `feature/<ชื่อ>` แยกจาก `main` ก่อนเริ่มเขียนโค้ดเสมอ ไม่ต้องใช้ดุลยพินิจ ไม่มีข้อยกเว้น** พัฒนา+เทสทั้งหมดบน branch นั้นจนเสร็จ (รวม syntax check, ทดสอบเล่นเกมจริง) แล้ว **ถามผู้ใช้ก่อนทุกครั้ง** ว่าจะ merge เข้า `main` ไหม รอคำยืนยันก่อนเสมอ ห้าม merge เข้า `main` เอง
- **Visitor badge ใน footer (`.visitor-badge` img src) — มี 2 จุด**: ใช้บริการ `hitscounter.dev` (เปลี่ยนจาก `visitor-badge.laobi.icu` ที่ปิดบริการไปแล้ว เมื่อ 2026-07-14) counter key คือ query param `url=` (URL-encoded) — หน้าหลัก `index.html` ใช้ `url=https%3A%2F%2Fowlkids.net` (prod) / `url=https%3A%2F%2Fowlkids.net%2Ftest` (dev), หน้าโหมดคุณครู `teacher/index.html` ใช้ `url=https%3A%2F%2Fowlkids.net%2Fteacher` (prod) / `url=https%3A%2F%2Fowlkids.net%2Fteacher-test` (dev) — param อื่น (`label=visitors`, `icon=eye`, `color=`) คงเดิมทั้งสองโหมด ระหว่างพัฒนา/ทดสอบใช้ค่า dev เสมอ (กันนับยอด visitor จริงเพี้ยนจากการเทส) แต่ก่อน push/deploy จริงทุกครั้งต้อง **แก้ param `url=` เป็นค่า prod ทั้ง 2 ไฟล์** ก่อน commit แล้วพอ push สำเร็จเสร็จสิ้นแล้ว ให้ **แก้กลับเป็นค่า dev ทันทีทั้ง 2 ไฟล์** (แก้ไฟล์ตรงๆ เลย **ไม่ต้อง commit หรือ push**) เพื่อกลับสู่โหมดพัฒนาปกติ — ทำทุกครั้งที่ push ไม่ต้องถามก่อน
- **ตรวจสอบก่อนบอกว่าทำเสร็จ**: รัน JS syntax check กับไฟล์ JS ทั้ง 3 ไฟล์ (เรียงตามลำดับโหลดจริง) ก่อนส่งมอบงานทุกครั้ง เช่น
  ```bash
  node -e "
  const fs = require('fs');
  const files = ['js/data.js','js/owl-messages.js','js/app.js'];
  const combined = files.map(f=>fs.readFileSync(f,'utf8')).join('\n');
  try { new Function(combined); console.log('OK JS syntax'); } catch(e){ console.error('ERROR:', e.message); }
  "
  ```
- **หลัง push/deploy สำเร็จทุกครั้ง**:
  1. สรุปสิ่งที่เปลี่ยนแปลงสั้นๆ ให้ผู้ใช้ทราบ
  2. เขียนสรุปนั้นลงไฟล์ `LOG.md` ด้วย โดยเพิ่มเป็นรายการใหม่ **ไว้บนสุด** ของไฟล์ (รูปแบบ: `- YYYY-MM-DD: สรุปสั้นๆ`) แล้วอัปเดตหัวข้อ "ประวัติการเปลี่ยนแปลง (changelog)" ใน `CLAUDE.md` ให้เป็นรายการล่าสุด 5 รายการเสมอ (ตัดรายการเก่าสุดออกถ้าเกิน 5) และถ้าเปลี่ยนแปลงพฤติกรรม/สถานะสำคัญของแอป ให้อัปเดตหัวข้อ "สถานะปัจจุบันของแอป" ใน `CLAUDE.md` ให้ตรงด้วย
  3. ก่อน push/deploy ทุกครั้ง ให้ตรวจและอัปเดต `README.md` ให้ตรงกับสถานะล่าสุดของแอปด้วย (เช่น ฟีเจอร์ใหม่, หมวดหมู่ใหม่) แล้ว commit `README.md` ไปพร้อมกับ `CLAUDE.md`, `LOG.md` และไฟล์โค้ดที่เปลี่ยนใน push เดียวกัน
  4. **ทุกครั้งที่ release version ใหม่ ต้องเขียนทับไฟล์ `changelog` ที่ root ด้วยเสมอ** (ไฟล์ text ธรรมดาไม่มีนามสกุลเหมือน `version`, ใช้แสดงในหน้า "รายละเอียดการอัพเดท" ของแอปให้ผู้ปกครองอ่าน) — **เก็บแค่ version ล่าสุด version เดียว เขียนทับของเดิมทุกครั้ง ไม่สะสมประวัติ** เขียนเป็นภาษาอ่านง่ายสำหรับผู้ปกครอง **ห้ามใส่ศัพท์ technical** (ชื่อไฟล์/ฟังก์ชัน/CSS class ฯลฯ) รูปแบบ: บรรทัดแรกคือเลข version, บรรทัดขึ้นต้น `## ` คือหัวข้อหมวด (เช่น เกมใหม่, ปรับปรุงหน้าตา, แก้ไขข้อผิดพลาด), บรรทัดขึ้นต้น `- ` คือรายการย่อยในหมวดนั้น (ดู parser `parseChangelog()` ใน `js/app.js`) แล้ว commit ไปพร้อมกับไฟล์อื่นๆ ใน push เดียวกัน
  5. **อัปเดต memory ถาวรของ Claude (directory memory ประจำโปรเจค) ทุกครั้งหลัง deploy เสร็จ** — ทบทวนว่ารอบงานนี้มีอะไรที่ควรจำข้ามเซสชันแล้วยังไม่ได้บันทึก เช่น บทเรียน/feedback จากผู้ใช้ระหว่างทำงานรอบนี้, ปัญหา deploy ที่เจอ+วิธีแก้ (เช่น Pages build ค้างต้องสั่ง rebuild), งานที่ยังค้างต่อเนื่อง — **ไม่ต้องบันทึกสิ่งที่ repo บันทึกไว้แล้ว** (โค้ด, LOG.md, CLAUDE.md, git history) ถ้าไม่มีอะไรใหม่จริงๆ ก็ข้ามได้ แต่ต้องทบทวนเสมอ
- **หลัง `git push` ทุกครั้ง ต้อง track สถานะ deploy ของ GitHub Pages แล้วสรุปสั้นๆ ให้ผู้ใช้ทราบ**:
  1. รอสักครู่แล้วเช็คด้วย `gh run list --limit 3` (workflow "pages build and deployment") และ/หรือ `gh api repos/Isarez/year1/pages` เพื่อดู `status`/`https_certificate`
  2. ถ้ายังไม่เสร็จ ให้เช็คซ้ำเป็นระยะ (เช่นทุก 60-180 วินาทีด้วย ScheduleWakeup) จนกว่าจะจบ — งานอื่นในเทิร์นนั้นทำต่อได้ระหว่างรอ
  3. เมื่อ deploy จบ ให้สรุปสั้นๆ 1-2 บรรทัดว่า **สำเร็จหรือไม่**:
     - สำเร็จ: ยืนยันด้วยการเช็ค `curl -I` ที่ URL จริงว่าได้ 200 และ `last-modified`/`etag` อัปเดตแล้ว (workflow run บางครั้งค้างสถานะ "queued" ทั้งที่ deploy จริงเสร็จแล้ว ให้ยึด `curl` เป็นหลักถ้าขัดแย้งกัน)
     - ไม่สำเร็จ: ต้องบอก **error message** ที่เจอจาก `gh run view <id> --log-failed` ให้ผู้ใช้ทราบด้วยเสมอ ไม่ใช่แค่บอกว่า "fail" เฉยๆ
  4. **แจ้งสรุปผล deploy (ข้อ 3) ใน Discord channel `1524252244965589082` ช่องนี้ช่องเดียวเท่านั้น** (ใช้ chat_id นี้กับ Discord reply tool) ไม่ว่าจะสั่ง push มาจากช่องทางไหนก็ตาม (Discord ช่องอื่น หรือ session ปกติ) — **ห้ามแจ้งผล deploy ซ้ำใน channel ที่สั่งงานมา** ให้แจ้งเฉพาะช่อง deploy log นี้ช่องเดียว
- **ถ้าเพิ่มเกม/หมวดใหม่ หรือแก้ไขอะไรที่มีนัยสำคัญ (breaking change, ปรับพฤติกรรมหลักของแอป, โครงสร้างไฟล์ใหม่ ฯลฯ) ต้อง `gh release create` ก่อน push/deploy เสมอ**:
  1. ตั้ง tag แบบ semver (เช่น `v1.1.0` ปรับ minor เมื่อเพิ่มฟีเจอร์/เกมใหม่, `v1.0.1` ปรับ patch เมื่อแก้บั๊ก/ปรับเล็กน้อย) — เช็ค tag ล่าสุดด้วย `git tag --list` หรือ `gh release list` ก่อนตั้งเวอร์ชันถัดไป
  2. เขียน release notes สรุปการเปลี่ยนแปลงเป็นภาษาไทย (ใช้เนื้อหาเดียวกับที่จะเขียนลง changelog ใน `LOG.md`)
  3. รัน `gh release create <tag> --title "<tag>: <สรุปสั้นๆ>" --notes "<รายละเอียด>"` แล้วค่อย push commit ตามปกติ (release อ้างอิง commit ปัจจุบันบน `main` ณ ตอนสร้าง)
  4. **แก้ไฟล์ `version` ที่ root ให้ตรงกับ tag ใหม่เสมอ** (เช่น `v1.3.0`) — ทำพร้อมกับ commit อื่นใน push เดียวกัน ห้ามลืม
  5. งานเล็กน้อย (ปรับ CSS ระยะห่าง, แก้ข้อความ, แก้สีเฉพาะจุด) ไม่ต้องสร้าง release ทุกครั้ง — ใช้ดุลยพินิจว่า "มีนัยสำคัญ" หรือไม่ ถ้าไม่แน่ใจให้ถามผู้ใช้ก่อน

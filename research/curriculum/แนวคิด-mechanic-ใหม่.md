# แนวคิด mechanic เกมใหม่ที่ควรพัฒนา (สังเคราะห์ข้ามวิชา)

ไฟล์นี้สรุปภาพรวมจากไอเดียโจทย์ในไฟล์รายชั้น/รายวิชาทั้ง 30 ไฟล์ (ป1-ป6 × 5 วิชา) โดยจัดกลุ่มเป็น "mechanic" (รูปแบบการเล่น) แทนที่จะแยกตามวิชา เพื่อดูว่า mechanic ไหนใช้ซ้ำได้ข้ามวิชา/ข้ามชั้นเยอะที่สุด — ยิ่งใช้ซ้ำได้เยอะ ยิ่งคุ้มที่จะลงทุนเขียน engine เพราะสร้างครั้งเดียวรองรับโจทย์ได้หลายร้อยข้อในหลายวิชา (แนวทางเดียวกับที่แอปทำกับ `ar-match`/`ar-match-en` ที่ใช้ engine เดียวกันแต่คนละคลังคำอยู่แล้ว)

## Tier 1 — ใช้ซ้ำได้แทบทุกวิชา ทุกชั้น (ควรทำก่อน ROI สูงสุด)

### 1. เรียงลำดับ (drag-to-order)
พบในไอเดียโจทย์กว่าครึ่งของไฟล์ทั้งหมด — เรียงจำนวน, เรียงเหตุการณ์ประวัติศาสตร์, เรียงขั้นตอนอวัยวะระบบย่อยอาหาร, เรียงวัฏจักรหิน, เรียงประโยคให้ได้ใจความ ฯลฯ
ต่อยอดจาก mechanic AR ต่อประโยค (`ar-thai`/`ar-eng`) ที่มีอยู่แล้วได้ตรงๆ แค่ generalize จาก "ลากคำเรียงเป็นประโยค" เป็น "ลากการ์ดใดๆ เรียงตามลำดับที่ถูกต้อง" (การ์ดจะเป็นคำ ตัวเลข รูปเหตุการณ์ หรือรูปอวัยวะก็ได้)

### 2. จับคู่ (matching)
เป็น mechanic ที่ปรากฏในไอเดียโจทย์เกือบทุกไฟล์ — จับคู่อวัยวะกับหน้าที่, รูปทรงกับชื่อ, ธงชาติกับประเทศ, ศัพท์อังกฤษกับภาพ, เหตุการณ์กับยุคสมัย
engine `ar-match`/`ar-match-en` ที่มีอยู่แล้วรองรับ non-emoji label ได้แล้วด้วย (ดู `matchLabelClass()` ฝั่งโหมดครู) แปลว่าขยายไปใช้กับคลังคำวิชาใหม่ทำได้โดยแทบไม่ต้องเขียน engine ใหม่ แค่เพิ่มคลังข้อมูลคู่จับคู่ใหม่

### 3. คัดแยกลงตะกร้า/หมวดหมู่ (sort into multiple bins)
ลากของ/คำไปใส่ตะกร้าตามหมวดที่ถูกต้อง เช่น สารอาหารให้พลังงาน/ไม่ให้พลังงาน, ของแข็ง/ของเหลว/แก๊ส, needs/wants, สัตว์บก/สัตว์น้ำ
ใกล้เคียง `ar-count` เดิม (ลากของใส่ตะกร้าเดียวให้ครบจำนวน) แต่ต่างตรงต้องมี **หลายตะกร้าพร้อมกัน** แล้วเช็คว่าของแต่ละชิ้นลงตะกร้าถูกหมวดหรือไม่ — เป็น mechanic ใหม่ที่ต้องออกแบบเพิ่ม ไม่ใช่แค่ขยายของเดิม

### 4. อ่านแผนภูมิ/กราฟ (pictograph/bar chart reading)
พบในคณิต (สถิติ), วิทยาศาสตร์ (ข้อมูลทดลอง), และสังคมศึกษา (งบประมาณ/สำรวจ) — โจทย์ลักษณะ "จากแผนภูมิ ใครมากที่สุด" ต้องมี component แสดงแผนภูมิรูปภาพ/แท่งง่ายๆ แล้วถามปรนัย เป็น UI component ใหม่ที่ใช้ซ้ำได้ทุกวิชา

## Tier 2 — เฉพาะบางวิชา แต่คุ้มเพราะใช้ข้ามหลายชั้นภายในวิชานั้น

### 5. แผนที่/พิกัด (map interaction)
สังคมศึกษาแทบทุกชั้น ตั้งแต่แผนผังห้องเรียน/บ้าน (ป.1-2) ไปจนถึงแผนที่จังหวัด (ป.4), ภูมิภาคของตนเอง (ป.5) และแผนที่ประเทศไทยทั้งประเทศ (ป.6) — ตรวจสอบกับเอกสารทางการแล้วพบว่าเนื้อหาระดับประถมเน้นที่ "ประเทศไทย" เป็นหลัก ยังไม่ถึงระดับอาเซียน/โลกเต็มรูปแบบ (นั่นเป็นเนื้อหาระดับมัธยม) ควรพิจารณาว่าจะต่อยอดจาก engine แผนที่ 3 มิติของ "บ้านของหนู" ได้หรือไม่ หรือทำเป็น 2D click-map ธรรมดาก็เพียงพอสำหรับระดับนี้

### 6. คูณ-หาร (multiplication/division drag-drop)
ต่อยอดจาก engine `ar-math`/`ar-math2`/`ar-math3` ตรงๆ ที่สุดในบรรดาทุก mechanic ใหม่ — แค่เปลี่ยนจากโจทย์ +/- เป็น ×/÷ และปรับช่วงตัวเลข เริ่มมีตั้งแต่คณิต ป.2

### 7. เศษส่วน/ทศนิยมแบบภาพ (fraction/decimal visual model)
คณิต ป.3 ขึ้นไป — ต้องมี component ใหม่ เช่น แถบ/วงกลมแบ่งส่วนให้ลากแรเงา หรือเส้นจำนวนให้วางจุด ไม่สามารถต่อยอดจาก mechanic เดิมได้ตรงๆ

## Tier 3 — เฉพาะจุด ซับซ้อนกว่าที่แอปเคยทำ ต้องออกแบบ mechanic ใหม่ทั้งหมด

### 8. ต่อวงจรไฟฟ้า (circuit builder)
วิทยาศาสตร์ ป.6 เท่านั้น — ลากถ่าน/สายไฟ/หลอดไฟมาต่อกันแล้วดูผล (ติด/ไม่ติด, อนุกรม/ขนาน) เป็น simulation mechanic ที่ซับซ้อนกว่าเกมที่มีอยู่ทั้งหมด

### 9. บล็อกคำสั่ง (block-based command builder)
เทคโนโลยี/การคิดเชิงคำนวณ (ว4.2) ปรากฏทุกชั้น ป.1-6 และเข้มข้นขึ้นเรื่อยๆ (ป.1-2 = ลำดับคำสั่งง่าย, ป.5-6 = ตัวแปร/เงื่อนไข/วนซ้ำแบบ Scratch) — พาตัวละครไปเป้าหมายด้วยการ์ดคำสั่งเดินหน้า/เลี้ยว/ทำซ้ำ เป็น mechanic ใหม่ทั้งหมด แต่คุ้มมากเพราะใช้ได้ทุกชั้นถ้าออกแบบให้ scale ความยากได้

### 10. อ่านจับใจความ (reading comprehension passage)
ภาษาไทย/อังกฤษ ป.4 ขึ้นไป — ต้องมี UI แสดงข้อความ/เรื่องสั้นยาวกว่าที่แอปเคยแสดง ตามด้วยปรนัยหลายข้อจากเรื่องเดียวกัน

### 11. ไวยากรณ์แบบลาก (grammar drag-drop)
ภาษาอังกฤษ ป.4 ขึ้นไป — subject-verb agreement, tense, ลากคำเติมประโยคให้ถูกไวยากรณ์ ต่างจาก `ar-eng` เดิม (แค่เรียงคำ) ตรงที่ต้องเช็ค "ความถูกต้องเชิงไวยากรณ์" ไม่ใช่แค่ลำดับคำที่กำหนดตายตัว

## หมายเหตุการออกแบบสำคัญ

- ทุก mechanic ใหม่ยังต้องคงหลักการเดิมของแอปตาม `CLAUDE.md` (static page ล้วน, ไม่มี build step/backend, localStorage เป็นฐานข้อมูล, ธีมการ์ตูนสีสดใสมนน่ารัก)
- เนื้อหา Tier 3 (อ่านจับใจความ, วงจรไฟฟ้า, บล็อกคำสั่ง) เริ่มเป็นนามธรรม/ซับซ้อนกว่าที่แอปเคยออกแบบมาทั้งหมด (ที่ผ่านมาเน้นเด็ก 5 ขวบ) — ก่อนเริ่มพัฒนาจริงควรทบทวนว่าธีม/ภาษาที่ใช้ยังเหมาะกับเด็ก ป.4-6 (10-12 ปี) หรือควรปรับโทนให้โตขึ้นเล็กน้อยสำหรับกลุ่มอายุนี้โดยเฉพาะ (ยังคงความเป็นการ์ตูนสดใสไว้ แต่ข้อความ/ความท้าทายอาจไม่ต้องเบสิกเท่าโหมดเด็กเล็ก)
- แนะนำเริ่มพัฒนาจาก **ป.1-2 ทุกวิชาก่อน** เพราะเนื้อหา/ความซับซ้อนใกล้เคียงกับสิ่งที่แอปมีอยู่แล้วมากที่สุด (ใช้ Tier 1 mechanic เป็นหลัก แทบไม่ต้องมี Tier 3) แล้วค่อยไล่ระดับขึ้นไปทีละชั้น แทนที่จะกระโดดไปทำเนื้อหา ป.5-6 ที่ต้องออกแบบ mechanic ใหม่จำนวนมากพร้อมกัน
- รายละเอียดตัวชี้วัดและไอเดียโจทย์เฉพาะเจาะจงแต่ละหัวข้อ ดูในไฟล์รายชั้น/รายวิชา (`ป1/` ถึง `ป6/`)

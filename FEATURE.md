# ฟีเจอร์ที่วางแผนไว้

ไฟล์นี้เก็บสเปคฟีเจอร์ที่ตกลงจะทำแล้ว (ต่างจาก [IDEA.md](IDEA.md) ที่เป็นไอเดียเกม/หมวดใหม่ที่ยังไม่แน่นอน) — รายละเอียดในนี้ควรชัดเจนพอเอาไปพัฒนาได้ทันที

**กติกา** (ดู [CLAUDE.md](CLAUDE.md) ด้วย):
- แต่ละฟีเจอร์เก็บเป็น 2 ส่วนเสมอ: **ชื่อฟีเจอร์** (สั้นๆ ใช้เป็น `##` heading) และ **รายละเอียด** (อธิบายกลไก/วิธีทำงานให้ชัดพอเอาไปทำต่อได้)
- เมื่อฟีเจอร์ไหนถูกพัฒนาเสร็จและ merge เข้าแอปจริงแล้ว ให้ลบออกจากไฟล์นี้ทันที
- ก่อนเริ่มพัฒนาฟีเจอร์ไหน ให้ถามผู้ใช้ก่อนเสมอว่า "จะทำฟีเจอร์นี้เลยไหม?" รอผู้ใช้ตอบ **"cf"** ก่อน แล้วอ่านรายละเอียดอีกครั้งอย่างละเอียด ถ้ามีจุดไหนไม่ชัดเจน/สงสัย/ขาดข้อมูลที่ต้องตัดสินใจ (เช่น "หมายเหตุ/ข้อจำกัดทางเทคนิค" ที่ระบุไว้ในฟีเจอร์นั้น) ให้ **ถามผู้ใช้ก่อนลงมือสร้าง** อย่าเดาเอง — ถ้าข้อมูลครบไม่มีข้อสงสัยแล้วให้เริ่มทำได้เลยโดยไม่ต้องถามซ้ำ

## รายการฟีเจอร์

## โหมดคุณครู (Teacher Mode) — เฟส 1
สถานะ: รอ "cf" ก่อนเริ่มเฟส 1

**จุดประสงค์:** ให้ผู้ปกครอง/คุณครูสร้างชุดโจทย์เองได้ แล้วให้เด็กเข้าเล่นด้วย mechanic เกมเดิมของแอป

**ทางเข้า:**
- ไม่มีปุ่มทางเข้าจากหน้าหลัก — เข้าผ่าน URL path ตรงๆ: `owlkids.net/teacher` (สร้างไฟล์ `teacher/index.html` ใน repo, GitHub Pages serve เป็น path อัตโนมัติ)
- ใช้ background + template + ธีมเดียวกับหน้าหลักทั้งหมด (reuse `css/style.css` ผ่าน relative path) — mechanic เกมทั้งหมดอ้างอิงจากหน้าหลัก
- หมายเหตุทางเทคนิค: `js/app.js` ผูกกับ DOM ของ `index.html` แน่นมาก → หน้า teacher ใช้ JS แยกของตัวเอง (เช่น `teacher/teacher.js`) ที่ copy/adapt เฉพาะ engine ที่ต้องใช้ ไม่โหลด `js/app.js` ตรงๆ

**Setup ครั้งแรก (ยังไม่เคยใช้งาน):**
- ฟอร์มใส่ ชื่อคุณครู + ชื่อโรงเรียน + เลือก avatar ของตัวเอง (ชุด avatar เดียวกับหน้าหลัก)
- เก็บใน localStorage (key แยกจากหน้าหลัก เช่น prefix `owlkids_teacher_`)

**หน้าหลักโหมดคุณครู (หลัง setup):**
- โชว์รายการโจทย์/ชุดเกมที่สร้างเอง แยกตามหมวดหมู่ที่สร้าง
- ถ้ายังไม่เคยสร้าง: ปุ่ม "เพิ่มโจทย์" ใหญ่กลางหน้าจอ / ถ้ามีแล้ว: ปุ่มเพิ่มโจทย์อยู่ตำแหน่งเหมาะสม (เช่น มุมบน)
- ชุดเกมที่ save draft (ยังไม่ publish) แสดงสถานะ draft แยกจากที่ publish แล้ว

**ฟอร์มสร้างเกม (กดเพิ่มโจทย์):**
- ระดับชั้น: อนุบาล 1-3, ประถม 1-6 (ใช้เป็นหมวดหมู่จัดกลุ่มในหน้าหลักโหมดคุณครู)
- เลือก mechanic เกม โดยอ้างอิงจากหน้าหลักว่ามีเกมแบบไหนบ้าง — **เฟส 1 เปิดใช้ mechanic "ถาม-ตอบปรนัย" (quiz) ก่อน** ตัวอื่น (จับคู่/ทายเงา/ฟังคำ ฯลฯ) โชว์ในตัวเลือกแบบ disabled "เร็วๆ นี้" เพราะโครงสร้างข้อมูลโจทย์ต่างกัน ต้องทำฟอร์มเฉพาะในเฟสถัดไป
- **เฟส 2 (ยืนยันกับผู้ใช้แล้ว 2026-07-10): เปิด mechanic AR** — แบบ "ลากการ์ดคำตอบใส่ช่อง" (สไตล์ ar-math) ใช้ข้อมูลโจทย์+คำตอบชุดเดียวกับ quiz ได้เลยไม่ต้องกรอกเพิ่ม, ส่วนต่อประโยค (พิมพ์ประโยคแล้วระบบตัดเป็นการ์ดคำ) และโยงเส้น (ใส่คู่ซ้าย↔ขวา) ต้องมีฟอร์มเฉพาะเพิ่ม — หมายเหตุ: engine AR (กล้อง+มือ MediaPipe) เป็นส่วนใหญ่สุดของแอป การ port ไปหน้า teacher ใช้เวลาพอสมควร
- ใส่ชื่อหัวข้อเกม
- เลือก logo เกม — เตรียม logo SVG ไว้ให้เลือกหลากหลาย (~12+ แบบ) ธีมเดียวกับ icon เกมหน้าหลัก (พาสเทล+เส้นขอบมน)
- ใส่จำนวนโจทย์ต่อรอบเล่น พร้อมหมายเหตุบนฟอร์ม: "ใส่โจทย์มากกว่าจำนวนนี้ได้ ระบบจะสุ่มมาให้ตามจำนวนที่เลือก"
- ช่องใส่โจทย์ + คำตอบ (เพิ่มได้หลายคำตอบต่อข้อ โดยระบุว่าข้อไหนคือคำตอบถูก) เพิ่มโจทย์ได้หลายๆ ข้อ
- checkbox: สุ่มลำดับโจทย์และสลับตำแหน่งคำตอบตอนเล่น
- ปุ่ม "save draft" (เก็บไว้แต่ยังไม่ publish) และ "publish" — publish แล้วโชว์บนหน้าหลักโหมดคุณครูให้เด็กกดเล่นได้

**การเล่น:**
- เข้าเล่นเกมที่สร้าง → โจทย์/คำตอบ/mechanic ตามที่สร้างไว้ (สุ่มโจทย์ตามจำนวนที่ตั้ง ถ้าเปิด shuffle ก็สลับตามนั้น)
- จบเกมไม่มีสติกเกอร์ (ไม่เหมือนหน้าหลัก) — แสดงผลคะแนน/ดาวได้ตามปกติ
- ข้อมูลทั้งหมด (โปรไฟล์ครู, ชุดเกม, draft) เก็บใน localStorage เท่านั้น ไม่มี backend

## บ้านของหนู (ตัวละคร 3D + ตกแต่งบ้าน สไตล์ Animal Crossing)
สถานะ: ยังไม่เริ่มทำ — รอ "cf" ก่อนเริ่มเฟส 1

**เทคโนโลยี 3D (ตัดสินใจแล้ว):**
- ใช้ Three.js **self-host** (โหลดไฟล์ minified เก็บไว้ใน repo เอง เช่น `js/vendor/three.min.js`, ไม่ใช้ CDN) เพื่อคงคุณสมบัติ static single-page site เดิม (เปิดผ่าน `file://` ได้, ไม่มี build step) — โหลดก่อน `app.js` ในลำดับ script tag
- ตัวละคร/เฟอร์นิเจอร์ทั้งหมด **สร้างด้วยโค้ด (procedural, ประกอบจากทรงเรขาคณิตพื้นฐาน: กล่อง/ทรงกระบอก/ทรงกลม/กรวย)** ไม่ใช่โหลดไฟล์โมเดล 3D สำเร็จรูป (.glb/.fbx) เพราะไม่มีเครื่องมือสร้างโมเดล 3D จริงให้
- **สไตล์ภาพที่ตัดสินใจ**: **blocky/voxel isometric-3D + flat/toon shading** (อ้างอิง `house_example/isomatic2d_style_1.png` เป็นทิศทางหลัก) ไม่ใช่การไล่เลียนแบบ Animal Crossing ของจริง (`animal_crossing_1.jpg`, `animal_crossing_2.jpg`) ตรงๆ — เพราะภาพตัวอย่าง Animal Crossing เป็น render คุณภาพระดับสตูดิโอเกม AAA (โมเดลออร์แกนิกโค้งมนละเอียด, PBR texture, soft shadow) ซึ่งเกินความสามารถของการเขียนโค้ดสร้างรูปทรงเรขาคณิตเอง ทำออกมาไม่มีทางใกล้เคียงและมีความเสี่ยงจะดูเพี้ยน/น่ากลัวสำหรับเด็กถ้าพยายามเลียนแบบความสมจริงแต่ทำได้ไม่ดี ในขณะที่แนว blocky/voxel (กล่อง/ทรงเรขาคณิตแบน สีสดใส เส้นขอบหนา) ทำได้จริงด้วย Three.js primitives + `MeshToonMaterial` (cel-shading) + เส้นขอบหนา (outline technique) ได้ผลลัพธ์สะอาด น่ารัก เข้ากับสไตล์ SVG icon แบนที่แอปใช้อยู่แล้ว และเบา/performant กว่ามากบนแท็บเล็ต
- ใช้ orthographic camera มุมเฉียงคงที่ (isometric-like) เพื่อให้ทรง block ดูเป็นระเบียบเหมือนภาพตัวอย่าง ไม่ใช้ perspective camera แบบเกมจริง
- กล้อง: fixed-angle มุมเฉียงคงที่คล้าย Animal Crossing ซูมเข้า-ออกได้ในช่วงจำกัด (ไม่ใกล้/ไกลเกินไป)

**หน้าใหม่:** view ใหม่ (ชื่อชั่วคราว `home-view`) เข้าถึงจากปุ่มใหม่ในหน้าเลือกโปรไฟล์เด็ก/หน้าแรก — ต้องไล่เพิ่มเข้า pattern ซ่อน/โชว์ view ทั้งหมดเหมือน view อื่นๆ (ดูหัวข้อ "หน้าเลือกหมวดมี 'โหมดเกม' 3 แบบ" ใน CLAUDE.md)

**แบ่งงานเป็น 4 เฟส (ต้องแจ้งผู้ใช้ทุกครั้งว่ากำลังทำเฟสไหนอยู่):**

1. **เฟส 1 — สร้างตัวละคร + นอกบ้าน**
   - หน้าสร้างตัวละคร (บังคับสร้างก่อนเข้าเกมครั้งแรก ถ้ายังไม่มีตัวละคร): เลือกผม, สีผม, ตา, สีตา, เพศ, เสื้อ, กางเกง/กระโปรง, รองเท้า — แต่ละอย่างมี **8-10 แบบ** preview ตัวละคร 3D หมุนดูได้ real-time ระหว่างเลือก
   - แผนที่นอกบ้าน: พื้นหญ้า, ต้นไม้, คลอง, สะพาน, บ้าน 1 หลังของเด็กคนนั้น
   - คลิกพื้นเพื่อเดินไปตำแหน่งนั้น (ถ้าติด element ให้เดินไปตำแหน่งที่ใกล้ที่สุดที่เดินได้แทน)
   - คลิกบ้าน → ตัวละครเดินไปหน้าบ้าน แล้วเปลี่ยนฉากเป็นมุมมองในบ้าน
   - ตัวละครมี animation เดิน (loop) และยืนเฉยๆ (idle) อย่างน้อย

2. **เฟส 2 — ภายในบ้านหลายห้อง**
   - แบ่งห้อง: ห้องนั่งเล่น, ห้องครัว, ห้องนอน, ห้องน้ำ (ผังห้องคงที่ต่อบ้าน 1 หลัง ไม่รองรับต่อเติม/เพิ่มห้องในเวอร์ชันแรก)
   - ระบบวางของแบบ **grid square block** ใช้ร่วมกันทั้งนอกบ้านและในบ้าน แต่ละห้อง/พื้นที่นอกบ้านมีขนาด grid คงที่ กำหนดโซนที่วางได้ (ห้ามทับกำแพง/ประตู/ทางเดิน/ของที่วางอยู่แล้ว)
   - เดินระหว่างห้องได้ (คลิกประตู/จุดเชื่อมห้อง)

3. **เฟส 3 — ระบบสะสม/ตกแต่ง**
   - เล่นเกม (quiz/AR/skill หมวดไหนก็ได้) จบ 1 รอบ → สุ่มเฟอร์นิเจอร์ **3 ชิ้นจากคลังที่ยังไม่เคยได้** ให้เด็กเลือก 1 ใน 3 (ได้ทุกครั้งที่เล่นจบเกม ไม่มีเงื่อนไขจำนวนดาว) — ชิ้นที่ได้แล้วจะไม่ถูกสุ่มซ้ำจนกว่าจะได้ครบคลัง
   - คลังเฟอร์นิเจอร์เริ่มต้น generate ไว้ในระบบ **ให้เยอะเพื่อความหลากหลาย** — เสนอ ~20-30 ชิ้นต่อชุด แยก 2 ชุด (นอกบ้าน เช่น รั้ว/ของประดับสนาม/ไฟ กับ ในบ้าน เช่น เตียง/โต๊ะ/ตู้/โซฟา) รวม ~40-60 ชิ้น
   - Panel แสดงเฟอร์นิเจอร์ที่เด็กมี (แยกชุดนอก/ใน) ลากจาก panel ไปวางบน grid ที่ว่างในฉากปัจจุบัน หยิบย้ายที่วางใหม่ได้ตลอด เก็บกลับ inventory ได้

4. **เฟส 4 — animation/polish**
   - ปรับท่าเดิน/ยืน/หมุนตัวให้ smooth (interpolation ระหว่าง pose หรือ `THREE.AnimationMixer`)
   - effect เล็กๆ ตอนได้เฟอร์นิเจอร์ใหม่ (sparkle/pop), transition ฉากเข้า-ออกบ้าน/ห้อง

**เก็บข้อมูล:** localStorage ทั้งหมด ต่อเด็ก 1 คน (ต่อยอดจากระบบ progress เดิม) — เพิ่ม state ใหม่: ตัวละคร (ชิ้นที่เลือกแต่ละส่วน), ตำแหน่งเฟอร์นิเจอร์ที่วางแล้วต่อห้อง/นอกบ้าน, คลังเฟอร์นิเจอร์ที่มี (id + ได้มาหรือยัง)

**ข้อควรระวัง/สิ่งที่ต้องทำระหว่าง implement:**
- ต้องรวมข้อมูลตัวละคร/บ้าน/เฟอร์นิเจอร์เข้าไปในระบบ export/import ย้ายข้อมูลข้ามอุปกรณ์เดิมด้วย ไม่งั้นย้ายเครื่องแล้วข้อมูลบ้านหาย
- Performance บนแท็บเล็ต/มือถือ ต้อง cap จำนวน poly/แสง/เงาไม่ให้หนักเกินไป (ใช้ `isMobileViewport()` เดิมช่วยตัดสินใจคุณภาพ)
- ต้องหาไฟล์ three.min.js มา self-host ก่อนเริ่มเฟส 1 (ดาวน์โหลดครั้งเดียวเก็บใน repo)
- ปุ่มเข้าหน้า "บ้านของหนู" วางตรงไหนของหน้าแรก — เสนอปุ่มใหม่คู่กับปุ่มเข้าเลือกหมวดเกม (ยังไม่ตัดสินใจ ถามตอนเริ่มเฟส 1)

- กดแล้วเปิด file picker ให้เลือกไฟล์จากเครื่องผู้ใช้
- อ่านไฟล์ → decode Base64 → **verify signature ก่อนเสมอ** — ถ้า signature ไม่ถูกต้อง/ไฟล์เสียหาย ให้ขึ้นแจ้งเตือนผู้ใช้ (ไฟล์ไม่ถูกต้อง ไม่สามารถนำเข้าได้) แล้วหยุด ไม่ import
- ถ้า signature ผ่าน → parse JSON → นำเข้าข้อมูลเด็ก (และ progress ของเด็กคนนั้น) ลง localStorage ของเครื่อง
- **render หน้าเลือกโปรไฟล์ใหม่ทันที** ให้เห็นเด็กที่เพิ่ง import เข้ามาปรากฏในรายการ
- เมื่อ import สำเร็จ ให้ถามผู้ใช้ว่า "ต้องการลบไฟล์ข้อมูลเด็กที่เพิ่ง import ออกจากเครื่องไหม?" — ถ้าผู้ใช้ confirm ให้ลบไฟล์นั้นออกจากเครื่องผู้ใช้

**ข้อตกลงที่ตัดสินใจแล้ว:**
- **การลบไฟล์หลัง import**: ไม่ทำ — ตัดออกจากสเปค ไม่มีแม้แต่ข้อความแนะนำ
- **Signature**: ใช้ checksum ธรรมดา + salt/prefix ที่เฉพาะของแอปนี้ เพื่อให้รู้ว่าไฟล์มาจาก logic ที่แอปสร้างขึ้น (ป้องกันนำไฟล์สุ่มจากที่อื่นมา import)
